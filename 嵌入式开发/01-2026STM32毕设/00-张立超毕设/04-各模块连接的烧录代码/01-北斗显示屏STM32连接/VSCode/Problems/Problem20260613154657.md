## 任务目标

调试当前 STM32 VS Code 工程，使其从 `main.c` 相关编译失败恢复到可以成功构建固件，并梳理后续烧录流程。

## 思考过程

先复现编译错误，再根据真实错误做最小修改。当前工程是 CMake + Ninja + ST 官方 VS Code 插件工具包，优先使用现有 `CMakePresets.json` 和 `build/Debug` 目录，不切换工程体系。

## 遇到的问题

初始目录枚举命令在当前工作区下超时，包括 `rg --files` 和目录列表命令。单个简单命令可以执行，因此判断应减少全目录扫描，改为读取关键文件和直接执行构建。

第一次执行 `cmake --build --preset Debug` 时失败，错误显示 `build/Debug/CMakeCache.txt` 记录的旧源码路径为 `F:/Note/...`，而当前工程路径为 `F:/Programme/Note to WebNote/Note/...`。这是移动或复制工程后常见的 CMake 缓存路径不一致问题，并不是 `main.c` 语法错误。

烧录阶段最初调用 `STM32_Programmer_CLI` 并使用 `mode=UR` 后返回：

```text
ST-LINK error (DEV_CONNECT_ERR)
```

随后用户反馈 STM32CubeProgrammer GUI 可以正常写入程序，说明硬件、ST-Link、芯片供电和基本 SWD 连接均正常。问题定位为 CLI 任务参数不匹配：`mode=UR` 是 under reset 连接模式，常用于需要通过 NRST 抢占连接的场景；当前普通 SWD 模式即可正常连接。

## 解决方法

先安全删除生成目录 `build/Debug`，删除前确认其解析路径位于当前工作区下，避免误删源码或其它目录。随后重新执行：

```text
cmake --preset Debug
cmake --build --preset Debug
```

重新配置和编译均成功。

同时修正会影响 VS Code 烧录/调试的配置：

- `openocd.cfg` 的目标芯片配置由空的 `target/.cfg` 改为 `target/stm32f1x.cfg`。
- `.vscode/launch.json` 的固件路径由不存在的 `./build/debug/projectName.elf` 改为实际生成的 `./build/Debug/VSCode.elf`。
- `.vscode/tasks.json` 从旧插件命令 `${command:stm32-for-vscode.*}` 改为显式 CMake 构建和 ST 官方 `STM32_Programmer_CLI` 烧录命令；烧录连接参数使用普通 `port=SWD`，不再强制 `mode=UR`。


## 任务目标
重新配置移动到 `VSCode2` 目录后的 STM32 工程，使其能够重新编译，并保留 VS Code 中构建和烧录任务的可用性。

## 思考过程
当前假设：目标芯片仍为 `STM32F103C8`，烧录方式仍为 ST-Link/SWD，工具链继续使用 STM32Cube 本地 bundle 中的 `arm-none-eabi-gcc`、Ninja 和 STM32_Programmer_CLI。

先检查项目类型和配置文件，确认工程包含 `CMakePresets.json`、`.vscode/tasks.json`、`.vscode/launch.json`、`openocd.cfg`，主要构建方式是 CMake + Ninja。没有先改源码，因为移动目录通常只会影响构建缓存或绝对路径配置。

## 遇到的问题
执行 `cmake --preset Debug` 失败。错误显示当前 `build/Debug/CMakeCache.txt` 是在旧目录 `.../VSCode/build/Debug` 下生成的，而当前源码目录已经变为 `.../VSCode2`。CMake 同时提示当前 `CMakeLists.txt` 与旧缓存记录的源码路径不一致。

工具路径检查结果正常：`arm-none-eabi-gcc.exe`、`ninja.exe`、`STM32_Programmer_CLI.exe` 和 `openocd.exe` 均存在。

## 解决方法
删除并重新生成 `build/Debug` 这个 CMake 生成目录。该目录是构建产物和缓存目录，不修改业务源码。

同时调整 `.vscode/tasks.json`：让 `Build STM` 和 `Build Clean STM` 任务依赖 `Configure STM`，这样在缓存被清理或工程移动后，VS Code 中直接点击构建也会先重新配置。

没有修改 `CMakeLists.txt`、源码文件或烧录目标配置，因为现有工程配置本身有效，失败原因集中在旧 CMake 缓存。

## 验证结果
`cmake --preset Debug` 已成功重新生成配置，生成目录更新为当前 `VSCode2/build/Debug`。

`cmake --build --preset Debug` 编译通过，生成 `build/Debug/VSCode.elf`。资源占用：

- RAM: `2264 B / 20 KB`，`11.05%`
- FLASH: `14940 B / 64 KB`，`22.80%`

使用 STM32CubeProgrammer 执行烧录命令：

```powershell
STM32_Programmer_CLI.exe -c port=SWD -w build\Debug\VSCode.elf -v -rst
```

检测到 ST-Link 序列号 `37FF71064E5734368D191B43`，目标芯片识别为 `STM32F101/F102/F103 Medium-density`，电压 `3.27V`。写入、校验均成功，并已执行软件复位。

CMake 重新配置时出现 Windows 路径长度警告，但本次完整编译已经通过，暂不额外改动构建目录结构。

## 代码变更摘要
- `.vscode/tasks.json`：为 `Build STM` 和 `Build Clean STM` 添加 `dependsOn: Configure STM`，避免移动目录或清理缓存后直接构建失败。
- `build/Debug`：删除旧目录并由 CMake 重新生成，清除旧 `VSCode` 路径缓存。
- `Problems/Problem20260613160746.md`：记录本次问题、处理过程和验证结果。

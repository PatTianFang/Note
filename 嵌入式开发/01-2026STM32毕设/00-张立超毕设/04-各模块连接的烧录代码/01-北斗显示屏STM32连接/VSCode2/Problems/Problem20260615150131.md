## 任务目标
排查 VS Code 中无法编译与调试的问题，并修复 Cortex-Debug 调试启动失败。

## 思考过程
从用户提供的 Cortex-Debug 日志看，GDB、OpenOCD、ELF 文件都能被启动或读取到一部分，说明工具链不是完全缺失。关键错误出现在 `target-download`：

```text
VSCode.elf: Invalid argument. (from target-download)
```

同时日志里的工程路径出现明显乱码，例如 `嵌入式开发` 被显示成 `宓屽叆...`。这说明 Cortex-Debug/GDB/OpenOCD 调试链路在处理中文路径时发生编码错误。命令行 `cmake --build --preset Debug` 返回 `ninja: no work to do`，说明当前代码本身可以编译，问题主要在 VS Code 调试下载阶段的路径编码。

修复思路：不移动原工程，创建一个英文路径目录联接 `F:\stm32_bds_vscode2` 指向当前工程，并让 VS Code 的构建、烧录、调试配置都通过该英文路径访问工程和 ELF，避开中文路径传给 GDB。

## 遇到的问题
当前工程真实路径包含中文目录和空格。Cortex-Debug 日志显示路径被错误解码，最终 GDB 执行 `target-download` 时无法打开 ELF，报 `Invalid argument`。

## 解决方法
已执行：

1. 创建英文路径目录联接 `F:\stm32_bds_vscode2` 指向当前工程。
2. 修改 `.vscode/tasks.json`，让 CMake 配置、构建和烧录任务在英文路径下执行。
3. 修改 `.vscode/launch.json`，让 Cortex-Debug 使用英文路径下的 `VSCode.elf` 和 `openocd.cfg`。
4. 清理并重新生成 `build/Debug`，确保 CMake 缓存和调试符号中的路径也尽量来自英文路径。

## 验证结果
从 `F:\stm32_bds_vscode2` 执行 `cmake --preset Debug` 成功，生成目录为：

```text
F:/stm32_bds_vscode2/build/Debug
```

执行 `cmake --build --preset Debug` 成功。资源占用：

- RAM: `1856 B / 20 KB`，`9.06%`
- FLASH: `12068 B / 64 KB`，`18.41%`

检查 `build/Debug/CMakeCache.txt`，`CMAKE_HOME_DIRECTORY`、`VSCode_SOURCE_DIR`、`VSCode_BINARY_DIR`、`CMAKE_TOOLCHAIN_FILE` 均已指向 `F:/stm32_bds_vscode2`。

使用 `F:\stm32_bds_vscode2\build\Debug\VSCode.elf` 执行 STM32CubeProgrammer 烧录，写入和校验均成功，并已执行软件复位。

VS Code 的 Cortex-Debug 需要用户在界面中重新启动一次 `Debug STM32` 验证。当前配置已经避开中文路径，预期可解决日志中的 `target-download Invalid argument`。

## 代码变更摘要
- `.vscode/tasks.json`：将配置、构建、清理构建、烧录任务的工作目录和 ELF 路径改为英文目录联接 `F:\stm32_bds_vscode2`。
- `.vscode/launch.json`：将 Cortex-Debug 的 `cwd`、`executable`、`configFiles` 改为英文路径，避免 GDB/OpenOCD 接收到中文乱码路径。
- `build/Debug`：删除旧缓存后重新生成，使 CMake 缓存路径切换到英文目录联接。
- `Problems/Problem20260615150131.md`：记录本次排查、原因和验证结果。

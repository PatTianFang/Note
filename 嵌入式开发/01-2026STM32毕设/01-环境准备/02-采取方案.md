# 02-采取方案：STM32CubeMX + VS Code + STM32Cube 插件 + STM32CubeProgrammer

> 更新时间：2026-05-30。  
> 本方案不使用第三方 STM32 Makefile 插件。  
> 核心思路：STM32CubeMX 负责生成工程，VS Code + ST 官方 STM32Cube 插件负责写代码和构建，STM32CubeProgrammer 负责把编译好的程序烧录进单片机。

## 1. 方案结论

本项目推荐采用下面这条路线：

```text
STM32CubeMX
  配置芯片、引脚、时钟、外设，生成 CMake 工程
        ↓
VS Code + ST 官方 STM32Cube 插件
  写代码、管理工程、构建程序、必要时启动调试
        ↓
生成 .elf / .hex / .bin
  编译产物，是真正要烧录到芯片里的程序文件
        ↓
STM32CubeProgrammer
  通过 ST-LINK 的 SWD 接口把程序写入 STM32 Flash
        ↓
STM32 开发板
  复位后从 Flash 运行程序
```

一句话理解：

```text
CubeMX 管配置，VS Code 写代码，官方插件管构建，CubeProgrammer 管烧录。
```

这比“所有事情都交给 VS Code 插件自动完成”更清晰，尤其适合小白排查问题。编译出错就看 VS Code，烧录出错就看 CubeProgrammer，问题边界比较清楚。

## 2. 每个工具负责什么

| 工具 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| STM32CubeMX | 选择芯片、配置引脚、配置时钟、配置外设、生成初始化代码 | 不负责长期写业务代码 |
| VS Code | 写代码、看代码、搜索、Git、Markdown、打开终端 | 本身不是 STM32 编译器 |
| STM32CubeIDE for Visual Studio Code | ST 官方 VS Code 扩展，负责 STM32 工程识别、CMake 构建、调试入口、工具链管理 | 不是桌面版 STM32CubeIDE |
| STM32CubeProgrammer | 烧录 `.elf`、`.hex`、`.bin`，擦除芯片，检查 ST-LINK 连接，查看 Option Bytes | 不负责写代码 |
| ST-LINK | 电脑和 STM32 之间的下载/调试硬件 | 不负责供电设计和代码生成 |
| STM32Cube MCU Package | 对应芯片系列的 HAL/LL 库、启动文件、例程 | 不负责自动完成业务功能 |

本项目如果使用 `STM32F103C8T6`，至少需要：

```text
STM32CubeF1
```

## 3. 思路

- 烧录流程更通用，不依赖某个 VS Code 插件。
- CubeProgrammer 是 ST 官方工具，排查 ST-LINK 连接问题更直接。
- `.elf`、`.hex`、`.bin` 都可以烧录，适合后期交付和备份。
- 即使 VS Code 配置坏了，只要已经编译出文件，仍然可以单独烧录。
- 后期换成 CubeIDE、Keil、CMake 命令行，只要能生成程序文件，烧录方法都一样。

- 编译和烧录分成两个动作。
- `.bin` 文件烧录时要知道 Flash 起始地址，常见 STM32 内部 Flash 起始地址是 `0x08000000`。

ST 官方 VS Code 插件会引导你安装或配置它需要的构建工具。不同版本界面可能略有差异，常见依赖包括：

| 工具 | 作用 |
| --- | --- |
| CMake | 生成构建系统 |
| Ninja | 执行构建 |
| Arm GNU Toolchain | 编译 STM32 程序 |
| STM32CubeCLT 或官方 Bundle | ST 命令行工具链集合或插件管理的工具包 |

新手建议：

- 先按 ST 官方插件提示安装 Bundle 或工具链。
- 如果插件提示缺少 CMake、Ninja、Toolchain，再按提示安装。
- 不要把工具链放在中文路径、空格很多的路径或很深的目录里。

## 6. 用 STM32CubeMX 创建 CMake 工程

下面以 `STM32F103C8T6` 为例。

### 6.1 选择芯片

打开 STM32CubeMX：

```text
Access to MCU Selector
```

搜索：

```text
STM32F103C8Tx
```

选中芯片后创建工程。

### 6.2 打开 SWD 调试接口

进入：

```text
System Core -> SYS
```

把 Debug 设置为：

```text
Serial Wire
```

原因：

- ST-LINK 使用 SWD 调试接口。
- 如果这里没开，后面可能出现能烧录但不能调试，甚至程序把调试引脚占掉。

### 6.3 配置时钟

如果你的 F103 小系统板有外部 8 MHz 晶振：

```text
System Core -> RCC -> High Speed Clock (HSE) -> Crystal/Ceramic Resonator
```

然后在 Clock Configuration 中把系统时钟配置到合适频率。常见 F103 可以到：

```text
72 MHz
```

注意：

- 必须以开发板实际硬件为准。
- 如果板子没有外部晶振，不要配置 HSE。
- 时钟配置错了，程序可能烧录成功但不运行。

### 6.4 配置一个验证功能

建议一开始只做两个简单验证。

LED 验证：

- 常见 F103C8T6 小系统板 LED 可能接在 `PC13`。
- 将 `PC13` 设置为 `GPIO_Output`。

串口验证：

- USART1 常用 `PA9`、`PA10`。
- `PA9`：USART1_TX。
- `PA10`：USART1_RX。
- 波特率先设置 `115200`。

### 6.5 选择 CMake 工程

进入：

```text
Project Manager -> Project
```

建议设置：

| 项目 | 推荐值 |
| --- | --- |
| Project Name | `F103_VSCode_CMake` |
| Project Location | `D:\STM32\Projects` |
| Toolchain / IDE | `CMake` |

本方案使用 ST 官方 VS Code 插件，所以推荐生成 CMake 工程。

### 6.6 生成代码

点击：

```text
GENERATE CODE
```

生成后通常会看到：

```text
F103_VSCode_CMake.ioc
CMakeLists.txt
Core
Drivers
startup_stm32f103xb.s
STM32F103C8Tx_FLASH.ld
```

这些文件的作用：

| 文件/目录 | 作用 |
| --- | --- |
| `.ioc` | CubeMX 配置文件 |
| `CMakeLists.txt` | CMake 构建入口 |
| `Core/Src` | 源文件 |
| `Core/Inc` | 头文件 |
| `Drivers` | HAL、CMSIS 等库 |
| `.s` | 启动文件 |
| `.ld` | 链接脚本，规定 Flash/RAM 地址 |

## 7. 在 VS Code 中打开和构建工程

### 7.1 打开工程

打开 VS Code：

```text
File -> Open Folder
```

选择工程根目录：

```text
D:\STM32\Projects\F103_VSCode_CMake
```

不要打开上一级目录：

```text
D:\STM32\Projects
```

原因：

- VS Code 插件需要在当前目录识别 `.ioc`、`CMakeLists.txt` 等工程文件。
- 打开错目录会导致插件找不到工程。

### 7.2 使用 ST 官方插件识别工程

打开 ST 官方 STM32Cube 插件视图，按界面提示执行：

```text
Discover / Import / Setup STM32Cube project
```

不同版本名称可能略有变化，原则是让插件识别当前 STM32 工程并配置 CMake 构建环境。

如果插件提示安装 Bundle、工具链、CMake 或 Ninja，按提示安装。

### 7.3 构建工程

构建方式通常有几种：

- 使用 ST 官方插件面板里的 Build。
- 使用 VS Code CMake Tools 的 Build。
- 在 VS Code 终端中执行 CMake 构建命令。

成功现象：

- 终端没有 `error`。
- 出现编译产物。
- 通常会生成 `.elf` 文件。
- 根据工程配置，也可能生成 `.hex` 或 `.bin`。

常见编译产物位置可能类似：

```text
build\Debug\F103_VSCode_CMake.elf
build\Debug\F103_VSCode_CMake.hex
build\Debug\F103_VSCode_CMake.bin
```

实际路径以你的 VS Code 构建输出为准。

## 8. 编译产物怎么选

| 文件类型 | 是否推荐烧录 | 是否需要手动填地址 | 说明 |
| --- | --- | --- | --- |
| `.elf` | 推荐 | 通常不需要 | 带调试信息，CubeProgrammer 可以识别地址 |
| `.hex` | 推荐 | 通常不需要 | 常用于发布和烧录，文件中包含地址信息 |
| `.bin` | 可以 | 需要 | 纯二进制文件，必须指定写入地址 |

新手建议：

- 优先烧录 `.elf` 或 `.hex`。
- 如果只有 `.bin`，地址填写：

```text
0x08000000
```

这是 STM32 内部 Flash 的常见起始地址。

## 9. 使用 STM32CubeProgrammer 图形界面烧录

这是最推荐小白先掌握的方法。

### 9.1 连接硬件

常见 ST-LINK SWD 接线：

```text
ST-LINK SWDIO -> STM32 SWDIO
ST-LINK SWCLK -> STM32 SWCLK
ST-LINK GND   -> STM32 GND
ST-LINK 3.3V  -> 目标板 3.3V，按开发板供电方式决定是否连接
ST-LINK NRST  -> STM32 NRST，可选但建议接
```

注意：

- GND 必须共地。
- 目标板必须供电。
- BOOT0 正常运行时应为低电平。
- USB 线必须是数据线，不是只能充电的线。

### 9.2 连接芯片

打开 STM32CubeProgrammer。

右侧选择：

```text
ST-LINK
Port: SWD
```

然后点击：

```text
Connect
```

连接成功后应能看到：

- Device ID。
- Flash size。
- 芯片连接状态。

如果连接失败，先不要管 VS Code，先把 CubeProgrammer 连接问题解决。

### 9.3 选择文件并烧录

进入：

```text
Erasing & Programming
```

选择编译得到的文件：

```text
.elf
.hex
.bin
```

如果选择 `.elf` 或 `.hex`：

- 通常不需要填写起始地址。

如果选择 `.bin`：

- 起始地址填写：

```text
0x08000000
```

建议勾选：

```text
Verify programming
Run after programming
```

然后点击：

```text
Start Programming
```

成功后会显示烧录完成，并复位运行。

## 10. 使用 STM32CubeProgrammer 命令行烧录

如果你想在 VS Code 终端里烧录，可以用 CubeProgrammer 的 CLI。

默认路径通常是：

```text
C:\Program Files\STMicroelectronics\STM32Cube\STM32CubeProgrammer\bin\STM32_Programmer_CLI.exe
```

### 10.1 烧录 `.elf`

```powershell
"C:\Program Files\STMicroelectronics\STM32Cube\STM32CubeProgrammer\bin\STM32_Programmer_CLI.exe" -c port=SWD -w ".\build\Debug\F103_VSCode_CMake.elf" -v -rst
```

### 10.2 烧录 `.hex`

```powershell
"C:\Program Files\STMicroelectronics\STM32Cube\STM32CubeProgrammer\bin\STM32_Programmer_CLI.exe" -c port=SWD -w ".\build\Debug\F103_VSCode_CMake.hex" -v -rst
```

### 10.3 烧录 `.bin`

`.bin` 必须写地址：

```powershell
"C:\Program Files\STMicroelectronics\STM32Cube\STM32CubeProgrammer\bin\STM32_Programmer_CLI.exe" -c port=SWD -w ".\build\Debug\F103_VSCode_CMake.bin" 0x08000000 -v -rst
```

### 10.4 擦除后再烧录

如果芯片状态混乱，可以先全片擦除：

```powershell
"C:\Program Files\STMicroelectronics\STM32Cube\STM32CubeProgrammer\bin\STM32_Programmer_CLI.exe" -c port=SWD -e all -w ".\build\Debug\F103_VSCode_CMake.hex" -v -rst
```

命令参数解释：

| 参数 | 含义 |
| --- | --- |
| `-c port=SWD` | 使用 ST-LINK 的 SWD 方式连接 |
| `-w` | 写入文件 |
| `0x08000000` | `.bin` 文件写入 Flash 的起始地址 |
| `-v` | 写入后校验 |
| `-rst` | 写入后复位 |
| `-e all` | 全片擦除 |

## 11. 在 VS Code 中做一个烧录任务

你可以把 CLI 命令写成 VS Code 任务，这样以后不用每次手打命令。

在工程目录创建或编辑：

```text
.vscode\tasks.json
```

示例任务：

```json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "STM32: Program ELF",
      "type": "process",
      "command": "C:\\Program Files\\STMicroelectronics\\STM32Cube\\STM32CubeProgrammer\\bin\\STM32_Programmer_CLI.exe",
      "args": [
        "-c",
        "port=SWD",
        "-w",
        "${workspaceFolder}\\build\\Debug\\F103_VSCode_CMake.elf",
        "-v",
        "-rst"
      ],
      "problemMatcher": []
    }
  ]
}
```

使用方式：

```text
Terminal -> Run Task -> STM32: Program ELF
```

注意：

- `F103_VSCode_CMake.elf` 要替换成你自己的文件名。
- 如果你的构建目录不是 `build\Debug`，也要改成真实路径。
- 第一次建议先用 CubeProgrammer 图形界面烧录，确认硬件连接没问题后再用任务自动化。

## 12. 调试和烧录的区别

很多新手会把调试和烧录混在一起。

| 操作 | 目的 | 是否会写入 Flash | 常用工具 |
| --- | --- | --- | --- |
| Build | 把代码编译成程序文件 | 不会 | VS Code、CMake、编译器 |
| Program/Download | 把程序写入芯片 Flash | 会 | STM32CubeProgrammer |
| Debug | 下载程序并进入断点调试 | 通常会 | ST 官方插件、调试器 |
| Run | 让芯片运行程序 | 不一定 | 调试器或复位 |

本方案建议：

- 平时验证固件是否能运行：用 CubeProgrammer 烧录。
- 需要看变量、单步执行、断点：用 VS Code 官方插件调试。
- 遇到连接问题：优先用 CubeProgrammer 排查。

## 13. 第一次验证

### 13.1 LED 闪烁

在 `while (1)` 的用户代码区加入：

```c
HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
HAL_Delay(500);
```

说明：

- `PC13` 只是常见 F103 小系统板 LED 引脚，必须以你的板子为准。
- 有些板子的 LED 是低电平亮。

验证流程：

1. VS Code 构建工程。
2. 找到 `.elf` 或 `.hex`。
3. 用 CubeProgrammer 烧录。
4. 勾选 `Run after programming`。
5. 观察 LED 是否闪烁。

### 13.2 串口打印

配置 USART1 后，可以发送：

```c
const uint8_t msg[] = "hello stm32\r\n";
HAL_UART_Transmit(&huart1, (uint8_t *)msg, sizeof(msg) - 1, 100);
HAL_Delay(1000);
```

串口工具设置：

```text
baud rate: 115200
data bits: 8
parity: none
stop bits: 1
flow control: none
```

接线：

```text
USB 转串口 TX -> STM32 RX
USB 转串口 RX -> STM32 TX
USB 转串口 GND -> STM32 GND
```

## 14. 常见问题

| 问题 | 常见原因 | 解决方法 |
| --- | --- | --- |
| CubeProgrammer 连不上芯片 | ST-LINK 驱动、接线、供电问题 | 检查设备管理器、USB 线、GND、SWDIO、SWCLK、目标板供电 |
| 提示 `No target connected` | SWD 接线错误、芯片没供电、SWD 频率过高 | 降低 SWD 频率，检查接线，必要时接 NRST |
| 烧录成功但程序不运行 | BOOT0 状态错误、时钟配置错误、代码卡死 | BOOT0 拉低，检查 HSE/PLL，先跑 LED 最小程序 |
| `.bin` 烧录后没反应 | 没填起始地址或地址错误 | `.bin` 地址填 `0x08000000` |
| `.elf` 文件找不到 | 工程没有构建成功或输出目录不同 | 在 VS Code 构建输出中找真实路径 |
| ST-LINK 被占用 | 其他软件正在连接 | 关闭 CubeProgrammer、调试会话或其他烧录工具后重试 |
| VS Code 能编译但不能烧录 | 烧录不属于 VS Code 默认能力 | 使用 CubeProgrammer GUI 或 CLI |
| 程序下载后调试不了 | CubeMX 没开 Serial Wire | 在 `SYS -> Debug` 设置 `Serial Wire` 后重新生成 |
| 串口乱码 | 波特率不一致或 GND 未共地 | 检查波特率、TX/RX、GND |

## 15. Git 管理建议

建议提交：

- `.ioc`
- `CMakeLists.txt`
- `Core`
- `Drivers`
- `.ld`
- `.s`
- 自己写的 `.c`、`.h`
- 必要的 `.vscode/tasks.json`
- 文档和记录

不建议提交：

- `build/`
- `Debug/`
- `Release/`
- `.elf`
- `.hex`
- `.bin`
- `.map`
- `.o`
- `.d`

`.gitignore` 示例：

```gitignore
build/
Debug/
Release/
*.elf
*.hex
*.bin
*.map
*.list
*.o
*.d
.vscode/ipch/
```

如果你把 VS Code 烧录任务提交到 Git，注意不要写死只有你电脑才存在的奇怪路径。CubeProgrammer 默认安装路径通常可以接受，但换电脑后仍可能不同。

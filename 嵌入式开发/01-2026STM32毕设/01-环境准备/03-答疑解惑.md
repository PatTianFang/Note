# 03-答疑解惑：STM32 开发环境里这些工具到底在干什么

> 更新时间：2026-05-30。  
> 本文回答当前阶段最容易混淆的 13 个问题。目标不是背概念，而是把“为什么要这样做”讲清楚。

## 0. 先看总流程

STM32 软件开发大概是下面这条链路：

```text
你选择芯片、配置引脚、配置时钟、配置外设
        ↓
STM32CubeMX 生成初始化代码和工程骨架
        ↓
你在 VS Code 里写 C 代码
        ↓
CMake 生成构建规则
        ↓
Ninja 或其他构建工具执行构建规则
        ↓
Arm GNU Toolchain 真正编译、汇编、链接
        ↓
生成 .elf / .hex / .bin
        ↓
STM32CubeProgrammer 通过 ST-LINK + SWD 烧录到 STM32 Flash
        ↓
STM32 从 Flash 启动运行程序
```

最容易混淆的一点：

```text
CMake 不是编译器。
VS Code 也不是编译器。
STM32CubeMX 也不是编译器。

真正把 C 语言变成机器码的，通常是 Arm GNU Toolchain 里的 arm-none-eabi-gcc。
```

## 1. 为什么要配置芯片、引脚、时钟、外设？STM32CubeMX 和 Keil 是什么关系？

### 1.1 为什么要配置

STM32 不是电脑软件，不能只写一句 `printf("hello")` 就自动知道硬件怎么工作。

单片机程序运行前，必须先告诉芯片：

- 哪些引脚当普通 GPIO。
- 哪些引脚当串口 TX/RX。
- 哪些引脚当 I2C、SPI、PWM、ADC。
- 系统主频是多少。
- 每个外设的时钟开没开。
- 中断开没开。
- 外设参数是多少，例如串口波特率、ADC 采样时间、定时器分频。

举例：

你想用 `PA9` 做串口发送，STM32 不会自动知道 `PA9` 是 USART1_TX。你必须配置：

- 打开 GPIOA 时钟。
- 打开 USART1 时钟。
- 把 `PA9` 设置为复用功能输出。
- 设置 USART1 的波特率、数据位、停止位、校验位。
- 如果使用中断，还要配置 NVIC。

如果不配置，常见结果是：

- 引脚没有输出。
- 串口没数据。
- 定时器不工作。
- I2C/SPI 没波形。
- 程序卡死。
- 下载后看起来“没反应”。

#### 1.1.1 哪些引脚当 GPIO，哪些当串口/I2C/SPI/PWM/ADC？这些功能分别是什么？

##### 功能名称中英对照

| 缩写 | 英文全称 | 中文 |
|------|---------|------|
| **TX** | Transmit | 发送 |
| **RX** | Receive | 接收 |
| **I2C** | Inter-Integrated Circuit | 串行通讯总线 |
| **SPI** | Serial Peripheral Interface | 串行外设接口 |
| **PWM** | Pulse Width Modulation | 脉冲宽度调制 |
| **ADC** | Analog-to-Digital Converter | 模数转换器 |
| **GPIO** | General Purpose Input/Output | 通用输入输出 |

##### 每个引脚能任意选择功能吗？

不能。这是由芯片**硬件设计**决定的，不是软件配置。

以 STM32F103 为例：

| 引脚 | 可用功能（硬件固定） |
|------|---------------------|
| PA9 | USART1_TX 或 TIM1_CH2 等 |
| PA10 | USART1_RX 或 TIM1_CH3 等 |
| PA5 | SPI1_SCK 或 ADC1_IN5 或 TIM2_CH1 |
| PB6 | I2C1_SCL 或 USART1_TX 或 TIM4_CH1 |
| PB7 | I2C1_SDA 或 USART1_RX 或 TIM4_CH2 |

也就是说，PA9 可以当 USART1_TX，但 PA9 **不能**当 I2C1_SCL（硬件不支持）。

##### 具体选哪个由什么决定？

**① 由芯片硬件决定（这个引脚天生能干啥）**

查数据手册的引脚定义表，确认该引脚支持哪些复用功能。

**② 由你的电路连接决定（原理图上这个引脚接到哪里）**

```
假设原理图上：
  MCU PA9 ───────────→ MAX232芯片 RX引脚

那么 PA9 必须配置为：USART1_TX
```

**③ 由外设需求决定（选对了才能正常工作）**

```
需要高速通信 → 选 SPI (几十MHz)
需要接多个设备 → 选 I2C (只接2根线)
需要输出精确电压 → 选 PWM + 滤波器
需要读取传感器电压 → 选 ADC
```

##### 一句话记住

> 引脚功能不是"想用啥就用啥"，而是"这个引脚天生能干啥"。硬件没有的功能，软件怎么配都没用。

#### 1.1.2 除了 GPIO/USART/I2C/SPI/PWM/ADC，还有什么？这些统称叫什么？

##### 统称：外设（Peripheral）

CPU 负责核心运算，外设负责和外部世界通信/控制。

**英文：** Peripheral

##### 常见外设完整分类

###### 通信类（最常用）

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| USART/UART | Universal Synchronous/Asynchronous Receiver Transmitter | 通用串口 | 调试串口、GPS、无线模块 |
| I2C | Inter-Integrated Circuit | I2C总线 | OLED屏幕、传感器、EEPROM |
| SPI | Serial Peripheral Interface | SPI总线 | Flash存储、屏幕、传感器 |
| CAN | Controller Area Network | CAN总线 | 汽车/工业网络通信 |
| USB | Universal Serial Bus | USB | 电脑通信、虚拟串口 |
| I2S | Inter-IC Sound | 音频总线 | 音频数据传输 |
| SDIO | Secure Digital Input/Output | SD卡接口 | 读取SD卡 |

###### 控制类

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| TIM | Timer | 定时器 | 定时、PWM输出、编码器、脉冲测量 |
| PWM | Pulse Width Modulation | 脉宽调制 | 电机调速、LED调光、舵机控制 |
| DAC | Digital to Analog Converter | 数模转换 | 输出模拟电压（音频、信号源） |
| DMA | Direct Memory Access | 直接内存访问 | 高速数据传输（省CPU资源） |

###### 采集类

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| ADC | Analog to Digital Converter | 模数转换 | 读取传感器电压、电池检测 |
| COMP | Comparator | 比较器 | 比较两个电压大小 |
| OPAMP | Operational Amplifier | 运算放大器 | 信号放大 |

###### 存储类

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| Flash | - | 内部Flash | 存储程序代码 |
| RAM | - | 内存 | 程序运行时存储变量 |
| EEPROM | - | EEPROM | 掉电保存数据 |
| FSMC/FMC | Flexible Static Memory Controller | 外部存储控制器 | 连接外部SRAM/NOR Flash |

###### 实时类

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| RTC | Real Time Clock | 实时时钟 | 计时、日期、闹钟 |
| WWDG | Window Watchdog | 窗口看门狗 | 程序异常检测复位 |
| IWDG | Independent Watchdog | 独立看门狗 | 程序异常检测复位 |

###### 特殊类

| 缩写 | 英文 | 中文 | 典型用途 |
|------|------|------|---------|
| CRC | Cyclic Redundancy Check | CRC校验 | 数据完整性校验 |
| RNG | Random Number Generator | 随机数生成器 | 产生随机数 |
| TS | Touch Sensing | 触摸检测 | 电容触摸按键 |
| ETH | Ethernet | 以太网 | 网络通信（部分型号） |
| DCMI | Digital Camera Interface | 摄像头接口 | 接入摄像头 |
| QSPI | Quad SPI | 四线SPI | 高速外部Flash |

##### 简化记忆

```
CPU（核心运算）
   │
   ├── 通信类：USART、I2C、SPI、CAN、USB
   ├── 控制类：TIM（PWM）、DAC
   ├── 采集类：ADC
   ├── 传输类：DMA
   ├── 存储类：Flash、RAM、EEPROM
   ├── 实时类：RTC、看门狗
   └── 特殊类：CRC、RNG、以太网、摄像头...

所有这些统称：外设（Peripheral）
```

##### 一句话记住

> **GPIO 是最简单的外设，其他都是特定功能的外设。F1 系列常用的大概 10 来个，实际项目中通常只会用到其中一部分。**

### 1.2 STM32CubeMX 是干什么的

STM32CubeMX 是 ST 官方提供的图形化配置工具。

它主要做这些事：

- 选择 STM32 芯片型号。
- 图形化选择引脚功能。
- 配置时钟树。
- 配置 GPIO、USART、I2C、SPI、ADC、TIM 等外设。
- 配置中断和 DMA。
- 生成初始化代码。
- 生成工程文件，例如 CMake、STM32CubeIDE、Keil 工程。

你可以把 CubeMX 理解成：

```text
硬件配置助手 + 初始化代码生成器
```

它不是主要写业务代码的地方。

### 1.3 Keil 是干什么的

Keil MDK 是一个传统 ARM 单片机 IDE。

它可以做：

- 写代码。
- 管理工程。
- 调用编译器。
- 下载程序。
- 单步调试。

你可以把 Keil 理解成：

```text
写代码 + 编译 + 下载 + 调试的一体化开发环境
```

### 1.4 CubeMX 和 Keil 的关系

CubeMX 和 Keil 不是同一类工具。

| 工具 | 主要角色 |
| --- | --- |
| STM32CubeMX | 配置芯片和生成初始化代码 |
| Keil MDK | 写代码、编译、下载、调试 |

CubeMX 可以生成 Keil 工程。

流程是：

```text
CubeMX 配置芯片和外设
        ↓
CubeMX 生成 MDK-ARM 工程
        ↓
Keil 打开 .uvprojx 工程
        ↓
Keil 编译、烧录、调试
```

所以它们是配合关系，不是简单的替代关系。

在我们当前方案里：

```text
CubeMX + VS Code + ST 官方插件 + CubeProgrammer
```

VS Code 和 ST 官方插件替代了 Keil 的大部分“写代码、构建、调试入口”功能。

## 2. CMake 工程是什么？CMake 是什么？

### 2.1 CMake 是什么

CMake 是一个“构建系统生成工具”。

这句话不好懂，可以换成大白话：

```text
CMake 不直接编译代码。
CMake 负责告诉其他工具：这个项目应该怎么编译。
```

它读取 `CMakeLists.txt`，然后生成 Ninja、Make、Visual Studio 等构建工具能理解的规则。

### 2.2 CMake 工程是什么

一个 CMake 工程通常包含：

```text
CMakeLists.txt
Core/Src/*.c
Core/Inc/*.h
Drivers
startup_xxx.s
xxx_FLASH.ld
```

其中最关键的是：

```text
CMakeLists.txt
```

它会描述：

- 项目名字。
- 使用 C 还是 C++。
- 芯片型号相关宏定义。
- 源文件有哪些。
- 头文件目录有哪些。
- 使用哪个编译器。
- 使用哪个链接脚本。
- 编译参数是什么。
- 链接参数是什么。
- 最后生成什么文件。

所以 CMake 工程不是一种新的 C 语言，而是一种描述“怎么把 C 代码编译成程序”的工程组织方式。

### 2.3 为什么 STM32 也能用 CMake

STM32 程序本质上也是 C 文件、汇编启动文件、头文件、链接脚本组成的工程。

只要把这些东西描述清楚，CMake 就可以生成构建规则。

例如：

```text
main.c
gpio.c
usart.c
startup_stm32f103xb.s
STM32F103C8Tx_FLASH.ld
HAL 库文件
CMSIS 头文件
```

CMake 负责把这些文件组织起来，让编译器知道：

- 编译哪些文件。
- 去哪里找头文件。
- 定义哪些宏。
- 链接到哪个地址。

### 2.3.1 在我们的方案里，CMake 不需要自己写

不需要。

在你的方案中：

```
CubeMX 配置芯片和外设
        ↓
点击 "Generate Code"
        ↓
自动生成：CMakeLists.txt + 所有源文件 + 链接脚本
        ↓
VS Code 打开工程
        ↓
ST 官方插件读取 CMakeLists.txt，调用 CMake 构建
```

**CubeMX 会自动生成完整的 CMakeLists.txt**，包含：

- 源文件列表
- 头文件路径
- 芯片型号相关宏定义
- 编译参数
- 链接脚本路径

#### 什么时候可能需要改？

| 场景 | 是否需要改 CMakeLists.txt |
|------|--------------------------|
| 正常使用 CubeMX 生成工程 | ❌ 不需要 |
| 添加一个新的 .c 文件 | CubeMX 重新生成会包含（不建议手动改） |
| 修改编译警告级别 | ⚠️ 可能需要（如加 `-Wall`） |
| 添加第三方库 | ⚠️ 可能需要手动添加 |

#### 总结

```
新手：完全不用碰 CMake
     ↓
CubeMX 生成 → 用就行
     ↓
等你遇到特殊需求（如添加外部库）→ 再学怎么改
```

> **大多数 STM32 项目用 CubeMX 生成的 CMake 工程就够了。**

## 3. 构建程序是不是编译？具体使用什么编译？CMake 和 C 语言、Makefile 是什么关系？

### 3.1 构建和编译的区别

日常口语里，很多人会把“构建”和“编译”混着说。

严格一点说：

```text
构建 Build > 编译 Compile
```

构建包含多个步骤：

```text
预处理 .c
  处理 #include、#define
        ↓
编译 .c -> .s 或 .o
  把 C 代码变成汇编或目标文件
        ↓
汇编 .s -> .o
  把汇编变成目标文件
        ↓
链接 .o -> .elf
  把多个目标文件合成最终程序
        ↓
转换 .elf -> .hex / .bin
  生成可烧录文件
```

所以：

- 编译是构建的一部分。
- 构建是完整生产程序文件的过程。

### 3.2 真正编译 C 代码的是谁

真正编译 STM32 C 代码的通常是：

```text
arm-none-eabi-gcc
```

它属于：

```text
Arm GNU Toolchain
```

常见工具包括：

| 工具 | 作用 |
| --- | --- |
| `arm-none-eabi-gcc` | 编译 C 代码，也可以驱动链接 |
| `arm-none-eabi-as` | 汇编 |
| `arm-none-eabi-ld` | 链接 |
| `arm-none-eabi-objcopy` | 把 `.elf` 转成 `.hex` 或 `.bin` |
| `arm-none-eabi-gdb` | 调试器后端 |

### 3.3 CMake 是编译器吗

不是。

CMake 的角色是：

```text
生成构建规则
```

编译器的角色是：

```text
把 C 代码变成机器码
```

举个比喻：

```text
CMake：写施工计划的人
Ninja：按施工计划调度工人干活的人
arm-none-eabi-gcc：真正拿工具加工零件的人
```

### 3.4 为什么要装 CMake

因为我们的方案使用 ST 官方 VS Code 插件和 CMake 工程。

CubeMX 生成 CMake 工程后，VS Code 需要通过 CMake 理解这个工程：

- 这个项目有哪些源文件。
- 头文件在哪里。
- 编译参数是什么。
- 链接脚本是什么。
- 输出文件在哪里。

没有 CMake，VS Code 插件就很难正确构建 CMake 工程。

### 3.5 CMake 和 C 语言是什么关系

C 语言是编程语言。

CMake 是构建配置语言。

它们不是同一个东西。

关系是：

```text
你用 C 语言写 STM32 程序。
CMake 描述这些 C 文件应该怎么编译。
```

### 3.6 CMake 和 Makefile 是什么关系

Makefile 是一种构建规则文件。

CMake 是一种生成构建规则的工具。

关系可以是：

```text
CMakeLists.txt
        ↓
CMake
        ↓
Makefile
        ↓
make 执行 Makefile
```

也可以是：

```text
CMakeLists.txt
        ↓
CMake
        ↓
Ninja 构建文件
        ↓
ninja 执行构建
```

所以：

- Makefile 可以手写。
- Makefile 也可以由 CMake 生成。
- CMake 不等于 Makefile。
- CMake 比 Makefile 更像上层工程描述。

在我们的方案里，更推荐：

```text
CubeMX 生成 CMake 工程
ST 官方 VS Code 插件调用 CMake
CMake 生成 Ninja 构建规则
Ninja 执行构建
Arm GNU Toolchain 编译
```

## 4. 调试需要连接单片机吗？需要烧录进去吗？为什么？

### 4.1 调试真实 STM32 时必须连接单片机

如果你要调试真实硬件，就必须连接单片机。

原因很简单：

```text
STM32 程序里面会访问真实硬件寄存器和外设。
这些硬件只存在于单片机内部，不存在于电脑里。
```

例如你写：

```c
HAL_GPIO_TogglePin(GPIOC, GPIO_PIN_13);
```

这行代码最终会操作 STM32 的 GPIO 寄存器。电脑上没有这个 GPIOC 寄存器，所以不能像普通 Windows 程序一样直接在电脑上运行。

### 4.2 调试时通常会先烧录

调试 STM32 时，常见流程是：

```text
编译生成 .elf
        ↓
调试器把程序下载到 STM32 Flash
        ↓
复位或暂停在 main
        ↓
你在 VS Code 里打断点、单步、看变量
```

为什么要烧录：

- STM32 需要从自己的 Flash 或 RAM 里取指令执行。
- 电脑只是控制它暂停、继续、读变量。
- 程序不进入芯片，芯片就没有代码可运行。

### 4.3 调试和普通烧录有什么区别

| 操作 | 目的 | 是否写入芯片 | 典型工具 |
| --- | --- | --- | --- |
| 烧录 | 把程序写入 Flash，让芯片独立运行 | 是 | STM32CubeProgrammer |
| 调试 | 写入程序后暂停、断点、单步、看变量 | 通常是 | ST-LINK + VS Code 插件 |

调试可以理解成：

```text
烧录 + 控制运行 + 观察内部状态
```

## 5. ST-LINK 的 SWD 接口是什么？为什么烧录要用 ST-LINK？能不能不用？为什么要装驱动？

### 5.1 ST-LINK 是什么

ST-LINK 是 ST 官方常用的下载/调试器。

它一头连接电脑 USB，另一头连接 STM32 的调试接口。

作用：

- 下载程序到 STM32。
- 单步调试。
- 打断点。
- 查看变量和寄存器。
- 擦除芯片。
- 读取芯片信息。

### 5.2 SWD 是什么

SWD 全称是 Serial Wire Debug。

它是 ARM Cortex-M 常用的两线调试接口。

常用信号：

| 信号 | 作用 |
| --- | --- |
| SWDIO | 调试数据线 |
| SWCLK | 调试时钟线 |
| GND | 共地 |
| 3.3V / VTref | 目标板电压参考或供电 |
| NRST | 复位线，可选但建议接 |

SWD 可以理解成：

```text
电脑通过 ST-LINK 和 STM32 内部调试模块说话的通道
```

### 5.3 为什么要用 ST-LINK

因为 STM32 出厂时并不像 U 盘那样插上 USB 就能直接拖文件。

你需要一种方式把程序写进芯片 Flash。

ST-LINK 的优势：

- 接线少。
- 稳定。
- 支持下载和调试。
- 不要求程序本身已经能运行。
- 即使你的代码写崩了，也通常还能重新连接和擦除。

### 5.4 ST-LINK 和单片机怎么连接

常见连接：

```text
ST-LINK SWDIO -> STM32 SWDIO
ST-LINK SWCLK -> STM32 SWCLK
ST-LINK GND   -> STM32 GND
ST-LINK 3.3V  -> 目标板 3.3V，按开发板供电方式决定
ST-LINK NRST  -> STM32 NRST，可选
```

为什么必须共地：

```text
电信号的高低电平必须有共同参考点。
没有 GND，共同语言就没了。
```

### 5.5 连接在一起有什么好处

连接 ST-LINK 后，你可以：

- 下载程序。
- 擦除芯片。
- 复位芯片。
- 暂停程序。
- 单步运行。
- 看变量。
- 看寄存器。
- 找程序卡在哪里。

这些是普通串口下载做不到或很难做到的。

### 5.6 能不能不用 ST-LINK

可以，但不推荐新手一开始不用。

可替代方式：

| 方式 | 说明 | 缺点 |
| --- | --- | --- |
| UART Bootloader | 通过芯片内置串口 Bootloader 下载 | 要设置 BOOT0，不能方便单步调试 |
| USB DFU | 部分芯片支持 USB 下载 | F103C8T6 常见小板不一定直接具备完整 USB DFU 条件 |
| J-Link | SEGGER 调试器 | 需要单独硬件和驱动 |
| 板载 Bootloader | 某些开发板自带 | 依赖开发板设计 |

结论：

```text
可以不用 ST-LINK，但用 ST-LINK 最适合入门和排错。
```

### 5.7 为什么 ST-LINK 要装驱动

ST-LINK 是一个 USB 设备。

Windows 需要驱动才能知道：

- 这个 USB 设备是什么。
- 应该用什么方式和它通信。
- 哪个程序可以访问它。

装好驱动后，STM32CubeProgrammer、CubeIDE、VS Code 插件才能找到 ST-LINK。

### 5.8 VS Code 插件会不会自动安装 ST-LINK 驱动

不要依赖 VS Code 插件自动安装驱动。

更稳的做法：

1. 安装 STM32CubeProgrammer。
2. 安装或确认 ST-LINK USB Driver。
3. 插上 ST-LINK。
4. 打开设备管理器确认识别正常。
5. 用 CubeProgrammer 先测试能否连接芯片。

## 6. 选择芯片、配置引脚、时钟、外设、生成初始化代码，如果自己干应该怎么做？

如果不用 CubeMX，自己做这些事情是可以的，但工作量很大。

### 6.1 你要先看哪些文档

至少要看：

- 芯片 Datasheet：看引脚、封装、电气参数。
- Reference Manual：看寄存器和外设说明。
- Programming Manual：看 Cortex-M 内核相关内容。
- 开发板原理图：看 LED、按键、晶振、模块接到哪些引脚。
- HAL/LL 文档或例程：看库函数怎么用。

### 6.2 自己配置 GPIO 大概要做什么

以 GPIO 为例，你要：

1. 打开 GPIO 外设时钟。
2. 设置引脚模式。
3. 设置输出速度。
4. 设置上拉/下拉。
5. 设置复用功能。
6. 操作输出数据寄存器。

如果是寄存器方式，你会直接写类似这些东西：

```c
RCC->APB2ENR |= RCC_APB2ENR_IOPCEN;
GPIOC->CRH &= ~(GPIO_CRH_MODE13 | GPIO_CRH_CNF13);
GPIOC->CRH |= GPIO_CRH_MODE13_1;
GPIOC->BSRR = GPIO_BSRR_BR13;
```

这只是一个 GPIO。串口、定时器、ADC 会更复杂。

### 6.3 自己配置时钟要做什么

你要理解：

- HSI 是什么。
- HSE 是什么。
- PLL 怎么倍频。
- AHB、APB1、APB2 分频。
- Flash 等待周期。
- 外设挂在哪条总线上。

时钟配置错了，可能导致：

- 程序不启动。
- 串口波特率不准。
- 定时器频率不对。
- USB 不工作。
- 系统不稳定。

### 6.4 自己生成初始化代码要做什么

你还要准备：

- 启动文件。
- 中断向量表。
- 链接脚本。
- 系统时钟初始化。
- HAL 初始化。
- 外设初始化函数。
- 中断服务函数。
- 编译工程配置。

CubeMX 的意义就是把这些重复、容易出错、但又必须做的事情自动化。

结论：

```text
高手可以自己写。
新手先用 CubeMX，后期再慢慢理解它生成了什么。
```

## 7. STM32CubeIDE for Visual Studio Code 插件有哪些作用？

这个插件是 ST 官方把 STM32 开发能力带到 VS Code 里的扩展。

主要作用：

- 创建或导入 STM32 工程。
- 识别 STM32CubeMX 生成的工程。
- 配置 CMake 构建环境。
- 管理 ST 官方工具链或 Bundle。
- 调用构建工具生成程序。
- 提供调试入口。
- 配合代码索引工具做跳转和补全。
- 提供 STM32 相关视图和工程操作入口。

它解决的问题是：

```text
VS Code 本身只是编辑器。
这个插件让 VS Code 更像 STM32 开发环境。
```

但要注意：

- 它不是 STM32CubeMX，不能完全替代 CubeMX 的图形化引脚/时钟配置。
- 它不是编译器，真正编译仍然依赖 Arm 工具链。
- 它不是 ST-LINK 驱动，驱动仍建议单独确认。
- 它不是 STM32CubeProgrammer，独立烧录仍建议用 CubeProgrammer。

## 8. STM32Cube MCU Package 有什么用？为什么一定要？

STM32Cube MCU Package 是某个 STM32 系列的官方软件包。

比如：

```text
STM32F103C8T6 -> STM32CubeF1
```

它通常包含：

| 内容 | 作用 |
| --- | --- |
| CMSIS 设备头文件 | 定义寄存器地址、中断号、芯片型号 |
| HAL 驱动 | ST 官方高级外设库，函数名常以 `HAL_` 开头 |
| LL 驱动 | 更底层的外设库，函数名常以 `LL_` 开头 |
| 启动文件 | 复位后程序从哪里开始，中断向量表在哪里 |
| 链接脚本模板 | Flash/RAM 地址和大小怎么分配 |
| 外设例程 | GPIO、USART、ADC、TIM 等参考代码 |
| BSP | 部分官方开发板的板级支持包 |

为什么需要它：

如果没有 MCU Package，CubeMX 和编译器会缺少很多关键东西。

常见缺失：

- 找不到 `stm32f1xx.h`。
- 找不到 `stm32f1xx_hal.h`。
- 找不到 `HAL_GPIO_Init`。
- 没有启动文件。
- 没有中断向量表。
- 没有外设驱动。
- CubeMX 无法正确生成对应系列的工程。

能不能完全不要？

可以，但你要自己提供这些东西：

- 自己写寄存器定义。
- 自己写启动文件。
- 自己写链接脚本。
- 自己写外设初始化。
- 自己写中断入口。
- 自己管理芯片差异。

这对初学者没有必要。

结论：

```text
用 STM32 官方生态开发，MCU Package 基本就是必须的地基。
```

## 9. CubeIDE 和 VS Code 有什么区别？

### 9.1 STM32CubeIDE

STM32CubeIDE 是 ST 官方的一体化 IDE。

特点：

- 基于 Eclipse。
- 集成 STM32 开发常用功能。
- 可以写代码、编译、下载、调试。
- 和 ST-LINK、CubeMX、CubeProgrammer 生态配合紧密。
- 新手配置成本低。

缺点：

- 界面比较重。
- 启动和索引可能慢。
- 写 Markdown、Git、通用编辑体验不如 VS Code 灵活。

### 9.2 VS Code

VS Code 本质是通用代码编辑器。

特点：

- 轻量。
- 插件生态丰富。
- 搜索、跳转、Git、Markdown 好用。
- 适合同时管理代码、文档、脚本。

缺点：

- 本身不懂 STM32。
- 需要安装插件和工具链。
- 出问题时要理解 CMake、工具链、烧录器之间的关系。

### 9.3 对比

| 对比项 | CubeIDE | VS Code |
| --- | --- | --- |
| 定位 | STM32 一体化 IDE | 通用代码编辑器 |
| 新手上手 | 更容易 | 配置更多 |
| 编译调试 | 内置集成较完整 | 依赖插件和工具链 |
| 文档/Git 体验 | 一般 | 强 |
| 适合场景 | 快速跑通 STM32 | 长期维护代码和文档 |
| 出问题排查 | 工具链集中 | 需要理解各工具分工 |

建议：

```text
第一次点灯可以用 CubeIDE。
毕业设计长期写代码和文档，可以用 VS Code 方案。
```

## 10. 为什么 `.bin` 烧录时要知道 Flash 起始地址？怎么知道是多少？

### 10.1 `.bin` 是什么

`.bin` 是纯二进制文件。

它只保存程序内容，不保存地址信息。

也就是说，`.bin` 文件只是在说：

```text
这些字节要写进芯片。
```

但它没有告诉烧录工具：

```text
从哪个地址开始写。
```

所以烧录 `.bin` 时必须手动指定地址。

### 10.2 `.hex` 和 `.elf` 为什么通常不用手动填地址

`.hex` 文件里带地址信息。

`.elf` 文件里也带地址、段信息、调试信息。

所以烧录工具能知道：

```text
这段内容应该写到 0x08000000。
那段内容应该写到别的位置。
```

### 10.3 STM32 Flash 起始地址怎么知道

方法 1：看芯片 Reference Manual 或 Datasheet 的 memory map。

方法 2：看链接脚本 `.ld`。

例如你可能会看到：

```ld
FLASH (rx) : ORIGIN = 0x08000000, LENGTH = 64K
RAM (xrw)  : ORIGIN = 0x20000000, LENGTH = 20K
```

这里：

```text
ORIGIN = 0x08000000
```

就是 Flash 起始地址。

方法 3：看 CubeMX 生成工程中的芯片配置和 linker script。

对常见 STM32 内部 Flash 程序来说，起始地址通常是：

```text
0x08000000
```

但不要死记。最终以你的芯片和链接脚本为准。

### 10.4 什么时候不是 0x08000000

有些情况程序不从默认 Flash 起始地址开始：

- 使用 Bootloader。
- 程序被分区，比如 Bootloader 在前，App 在后。
- 外部 Flash 启动。
- 特殊芯片或特殊存储映射。

例如 Bootloader 占用前 16 KB，应用程序可能从：

```text
0x08004000
```

开始。

所以烧录 `.bin` 时一定要确认链接脚本中的 `FLASH ORIGIN`。

## 11. ST-LINK USB Driver 是啥？

ST-LINK USB Driver 是 Windows 识别 ST-LINK 的驱动。

你可以把它理解成：

```text
电脑和 ST-LINK 之间的翻译层。
```

没有它，可能出现：

- 设备管理器显示未知设备。
- CubeProgrammer 找不到 ST-LINK。
- VS Code 插件找不到调试器。
- 能供电但不能下载。

安装后，Windows 才能正确识别 ST-LINK，并允许 ST 官方工具访问它。

建议验证方式：

1. 插入 ST-LINK。
2. 打开设备管理器。
3. 查看是否出现 ST-LINK 相关设备。
4. 打开 STM32CubeProgrammer。
5. 选择 ST-LINK + SWD。
6. 点击 Connect。

如果 CubeProgrammer 能连接目标芯片，驱动和基本硬件连接大概率没问题。

## 12. CMake、Ninja、Arm GNU Toolchain 分别干什么？

这三个经常一起出现，但角色完全不同。

| 工具 | 作用 | 类比 |
| --- | --- | --- |
| CMake | 生成构建规则 | 写施工方案 |
| Ninja | 执行构建规则 | 调度工人按步骤施工 |
| Arm GNU Toolchain | 真正编译、汇编、链接 | 真正切割、焊接、组装零件的工具 |

### 12.1 CMake

CMake 读取：

```text
CMakeLists.txt
```

然后生成 Ninja 可以执行的构建文件。

它关心：

- 源文件有哪些。
- 头文件路径有哪些。
- 编译参数有哪些。
- 链接脚本在哪里。
- 输出文件叫什么。

### 12.2 Ninja

Ninja 是一个构建执行器。

它不理解你的芯片原理，也不关心 GPIO。

它只按规则执行命令：

```text
编译 main.c
编译 gpio.c
编译 usart.c
汇编 startup.s
链接生成 .elf
转换生成 .hex/.bin
```

Ninja 的特点是快，适合配合 CMake。

### 12.3 Arm GNU Toolchain

Arm GNU Toolchain 才是真正处理 C 代码的工具链。

核心命令：

```text
arm-none-eabi-gcc
```

为什么名字这么长：

- `arm`：目标架构是 ARM。
- `none`：不是某个桌面操作系统。
- `eabi`：使用嵌入式 ABI 约定。
- `gcc`：GNU C Compiler。

它编译出来的程序不是给 Windows 跑的，而是给 ARM Cortex-M 单片机跑的。

### 12.4 三者配合关系

```text
CMakeLists.txt
        ↓
CMake 生成 Ninja 构建规则
        ↓
Ninja 执行构建命令
        ↓
Arm GNU Toolchain 编译/链接
        ↓
生成 .elf/.hex/.bin
```

所以：

```text
装 CMake 是为了理解和生成工程构建规则。
装 Ninja 是为了执行这些规则。
装 Arm GNU Toolchain 是为了真正编译 STM32 程序。
```

## 13. 外部晶振是什么？怎么看有没有？频率是多少？

### 13.1 晶振是什么

晶振是给单片机提供稳定时钟的器件。

STM32 运行需要时钟。没有时钟，CPU 和外设都无法按节奏工作。

时钟可以来自：

| 时钟源 | 说明 |
| --- | --- |
| HSI | 内部高速 RC 振荡器，芯片内部自带 |
| HSE | 外部高速晶振或外部时钟 |
| LSI | 内部低速 RC，常用于看门狗 |
| LSE | 外部 32.768 kHz 低速晶振，常用于 RTC |

### 13.2 外部晶振有什么用

外部晶振通常比内部 RC 更准确、更稳定。

它常用于：

- 提供准确系统时钟。
- 通过 PLL 倍频到更高主频。
- 提高串口、定时器等外设时钟准确性。
- USB、以太网、音频等对时钟精度要求较高的场景。
- RTC 使用 32.768 kHz 晶振保持时间。

### 13.3 HSE 和 LSE 的区别

| 类型 | 常见频率 | 用途 |
| --- | --- | --- |
| HSE | 8 MHz、12 MHz、16 MHz、25 MHz 等 | 系统主时钟来源 |
| LSE | 32.768 kHz | RTC 实时时钟 |

F103 小系统板常见：

- HSE：8 MHz。
- LSE：32.768 kHz，不一定有。

但必须以你的板子为准。

### 13.4 怎么看开发板有没有外部晶振

方法 1：看原理图。

搜索：

```text
HSE
OSC_IN
OSC_OUT
LSE
PC14
PC15
32.768K
8MHz
```

方法 2：看板子实物。

常见外观：

- 金属壳小长方体，标着 `8.000`、`12.000`、`16.000`。
- 小圆柱表晶，常见标称 `32.768 kHz`。

方法 3：看芯片引脚附近。

F103 的外部高速晶振通常接在：

```text
OSC_IN / OSC_OUT
```

低速 RTC 晶振通常接在：

```text
PC14 / PC15
```

方法 4：看商家资料或开发板说明。

很多 F103C8T6 小系统板资料会写：

```text
8M crystal
32.768K crystal
```

### 13.5 怎么知道频率是多少

优先级从高到低：

1. 看原理图标注。
2. 看开发板资料。
3. 看晶振外壳丝印。
4. 看 BOM。
5. 用示波器测量。

不建议用普通万用表测晶振频率。

### 13.6 CubeMX 里应该怎么选

如果板子有外部高速晶振：

```text
RCC -> HSE -> Crystal/Ceramic Resonator
```

如果板子没有外部高速晶振：

```text
RCC -> HSE -> Disable
```

如果外部有源时钟输入，而不是普通晶振：

```text
RCC -> HSE -> Bypass Clock Source
```

注意：

```text
Crystal/Ceramic Resonator 和 Bypass Clock Source 不是一回事。
```

普通两脚晶振用 `Crystal/Ceramic Resonator`。  
外部已经产生好方波时钟再输入 STM32，才用 `Bypass Clock Source`。

### 13.7 配错会怎样

如果你在 CubeMX 里设置了 HSE，但板子实际上没有外部晶振：

- 程序可能卡在时钟初始化。
- LED 不闪。
- 串口没有输出。
- 看起来像“烧录成功但不运行”。

如果频率填错：

- 系统主频计算错误。
- 串口波特率不准。
- 定时器时间不准。
- USB 等外设可能异常。

所以时钟配置一定要和硬件一致。

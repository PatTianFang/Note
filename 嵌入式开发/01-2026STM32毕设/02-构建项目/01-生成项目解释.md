# 01-生成项目解释：CubeMX 默认工程里每个文件的关系和作用

> 更新时间：2026-05-30。  
> 本文解释 `02-构建项目/example` 这个由 STM32CubeMX 生成的默认项目。  
> 这个工程不是 CMake 工程，而是 CubeMX 生成的 `EWARM V8.32` 工程，也就是给 IAR Embedded Workbench 使用的工程。

## 1. 先看这个工程是什么

`example` 是一个 STM32CubeMX 生成的默认工程。

从 `example.ioc` 可以看出：

| 项目 | 内容 |
| --- | --- |
| MCU | `STM32F103C8Tx` |
| 系列 | `STM32F1` |
| 封装 | `LQFP48` |
| 固件包 | `STM32Cube FW_F1 V1.8.7` |
| 工具链 | `EWARM V8.32` |
| 已配置外设 | `SYS`、`RCC`、`USART1`、`NVIC` |
| 调试接口 | `Serial Wire`，即 SWD |
| 串口引脚 | `PA9 = USART1_TX`，`PA10 = USART1_RX` |
| 调试引脚 | `PA13 = SWDIO`，`PA14 = SWCLK` |
| 系统时钟 | HSI 8 MHz，无 PLL |
| USART1 参数 | 115200，8N1，无流控 |
| 栈大小 | `0x400` |
| 堆大小 | `0x200` |

这类默认工程的意义是：

```text
CubeMX 根据 .ioc 配置
        ↓
生成 Core 用户代码框架
        ↓
复制 Drivers 官方库
        ↓
生成 EWARM 工程文件
        ↓
IAR 根据工程文件编译、链接、下载、调试
```

## 2. 工程总结构

`example` 的主要结构：

```text
example/
├─ example.ioc
├─ .mxproject
├─ Core/
│  ├─ Inc/
│  │  ├─ main.h
│  │  ├─ stm32f1xx_hal_conf.h
│  │  └─ stm32f1xx_it.h
│  └─ Src/
│     ├─ main.c
│     ├─ stm32f1xx_hal_msp.c
│     ├─ stm32f1xx_it.c
│     └─ system_stm32f1xx.c
├─ Drivers/
│  ├─ CMSIS/
│  └─ STM32F1xx_HAL_Driver/
└─ EWARM/
   ├─ Project.eww
   ├─ example.ewp
   ├─ example.ewd
   ├─ startup_stm32f103xb.s
   ├─ stm32f103xb_flash.icf
   └─ stm32f103xb_sram.icf
```

可以先按 5 类理解：

| 类别 | 目录/文件 | 作用 |
| --- | --- | --- |
| CubeMX 配置 | `example.ioc` | 保存芯片、引脚、时钟、外设配置 |
| 生成记录 | `.mxproject` | CubeMX 用来记录生成过哪些文件 |
| 用户工程代码 | `Core` | 你的主程序、中断、初始化框架 |
| 官方库 | `Drivers` | CMSIS、HAL、LL 等官方底层代码 |
| IAR 工程 | `EWARM` | IAR 的工程、调试、启动、链接配置 |

## 3. 一句话理解每一层

```text
example.ioc
  是 CubeMX 的源配置文件。

Core
  是你主要看、主要改的应用代码层。

Drivers
  是 ST 官方库，提供芯片头文件、HAL 函数、CMSIS 内核支持。

EWARM
  是 IAR 工程层，告诉 IAR 编译哪些文件、怎么链接、怎么调试。
```

不要一开始就被 `Drivers` 里大量文件吓到。当前工程真正编译的只是其中一部分，其他很多文件只是库文件被复制进来了，不一定都参与构建。

## 4. 整个程序从上电到 main 的运行流程

这个流程非常重要。

STM32 复位后不是直接从 `main.c` 第一行开始。

真实流程大致是：

```text
芯片复位
        ↓
读取向量表第 1 项：初始栈顶地址
        ↓
读取向量表第 2 项：Reset_Handler 地址
        ↓
进入 startup_stm32f103xb.s
        ↓
调用 SystemInit()
        ↓
IAR 运行库初始化全局变量、清零 bss
        ↓
进入 main()
        ↓
HAL_Init()
        ↓
SystemClock_Config()
        ↓
MX_GPIO_Init()
        ↓
MX_USART1_UART_Init()
        ↓
while (1) 主循环
```

对应文件关系：

| 阶段 | 文件 |
| --- | --- |
| 复位入口、向量表 | `EWARM/startup_stm32f103xb.s` |
| 早期系统初始化 | `Core/Src/system_stm32f1xx.c` |
| 主函数和外设初始化 | `Core/Src/main.c` |
| HAL 全局 MSP 初始化 | `Core/Src/stm32f1xx_hal_msp.c` |
| 中断处理 | `Core/Src/stm32f1xx_it.c` |
| 链接地址和内存分配 | `EWARM/stm32f103xb_flash.icf` |

## 5. `example.ioc`：CubeMX 的工程配置源头

`example.ioc` 是 CubeMX 最重要的文件。

它记录了：

- 选择的芯片。
- 引脚功能。
- 外设参数。
- 时钟配置。
- NVIC 中断配置。
- 工程名称。
- 生成工具链。
- 固件包版本。
- 栈和堆大小。

这个工程中关键配置：

```text
Mcu.CPN=STM32F103C8T6
Mcu.UserName=STM32F103C8Tx
ProjectManager.TargetToolchain=EWARM V8.32
ProjectManager.FirmwarePackage=STM32Cube FW_F1 V1.8.7
PA9.Signal=USART1_TX
PA10.Signal=USART1_RX
PA13.Signal=SYS_JTMS-SWDIO
PA14.Signal=SYS_JTCK-SWCLK
USART1.VirtualMode=VM_ASYNC
RCC.APB1Freq_Value=8000000
RCC.APB2Freq_Value=8000000
```

你可以把 `.ioc` 理解成：

```text
CubeMX 用来重新生成项目的配置数据库。
```

修改外设配置时，应该优先打开 `.ioc`，不要直接手改所有生成代码。

## 6. `.mxproject`：CubeMX 的生成记录

`.mxproject` 是 CubeMX 自己用的记录文件。

它记录：

- 上次生成了哪些头文件。
- 上次生成了哪些源文件。
- 使用了哪些 HAL 库文件。
- IAR 工程里包含哪些文件。
- include path 和宏定义。

你通常不需要手动修改它。

它的意义是：

```text
帮助 CubeMX 知道哪些文件是它管理的，重新生成时怎么更新工程。
```

新手建议：

- 不要手动编辑 `.mxproject`。
- 不要删除它。
- 如果工程要交给别人，保留它。

## 7. `Core`：你最应该关注的代码目录

`Core` 是用户工程代码目录。

它分为：

```text
Core/Inc  头文件
Core/Src  源文件
```

默认工程里，`Core` 文件不多，但每个都很关键。

### 7.1 `Core/Src/main.c`

`main.c` 是应用程序主入口。

它包含：

- `main()` 主函数。
- `SystemClock_Config()` 时钟配置。
- `MX_GPIO_Init()` GPIO 初始化。
- `MX_USART1_UART_Init()` USART1 初始化。
- `Error_Handler()` 错误处理。
- `while (1)` 主循环。

当前 `main()` 的核心流程：

```c
HAL_Init();
SystemClock_Config();
MX_GPIO_Init();
MX_USART1_UART_Init();

while (1)
{
}
```

每一步含义：

| 调用 | 作用 |
| --- | --- |
| `HAL_Init()` | 初始化 HAL 库、SysTick、底层 MSP |
| `SystemClock_Config()` | 设置系统时钟，这里使用 HSI 8 MHz |
| `MX_GPIO_Init()` | 打开 GPIOA 时钟 |
| `MX_USART1_UART_Init()` | 配置 USART1，115200 8N1 |
| `while (1)` | 主循环，你的业务代码最终会跑在这里 |

新手最常写代码的位置：

```c
/* USER CODE BEGIN 2 */
// 初始化完成后运行一次的代码
/* USER CODE END 2 */

while (1)
{
  /* USER CODE BEGIN 3 */
  // 循环执行的业务代码
  /* USER CODE END 3 */
}
```

不要把自己的代码写在没有 `USER CODE` 标记保护的区域，否则 CubeMX 重新生成时可能覆盖。

### 7.2 `Core/Inc/main.h`

`main.h` 是 `main.c` 的公共头文件。

它主要做：

- 包含 `stm32f1xx_hal.h`。
- 声明 `Error_Handler()`。
- 放全局宏定义。
- 放用户需要导出的类型、宏、函数声明。

它不是写主逻辑的地方。

适合放：

```c
#define LED_Pin GPIO_PIN_13
#define LED_GPIO_Port GPIOC
```

当前默认工程没有配置 LED，所以 `main.h` 中没有实际外设宏。

### 7.3 `Core/Src/stm32f1xx_hal_msp.c`

`msp` 是 MCU Support Package 的意思。

这个文件负责“HAL 外设初始化背后的底层硬件资源配置”。

比如 `main.c` 里调用：

```c
HAL_UART_Init(&huart1);
```

HAL 内部会回调：

```c
HAL_UART_MspInit(&huart1);
```

然后 `stm32f1xx_hal_msp.c` 里会真正配置：

- USART1 外设时钟。
- GPIOA 时钟。
- PA9 为复用推挽输出。
- PA10 为输入。

当前工程里的 USART1 MSP 初始化：

```c
__HAL_RCC_USART1_CLK_ENABLE();
__HAL_RCC_GPIOA_CLK_ENABLE();

PA9  -> USART1_TX
PA10 -> USART1_RX
```

还包含：

```c
HAL_MspInit()
```

这个函数在 `HAL_Init()` 期间调用，当前做了：

- 打开 AFIO 时钟。
- 打开 PWR 时钟。
- 关闭 JTAG，保留 SWD。

关键代码含义：

```c
__HAL_AFIO_REMAP_SWJ_NOJTAG();
```

意思是：

```text
禁用 JTAG，保留 SWD 调试接口。
```

这样可以释放部分 JTAG 引脚，但仍能用 ST-LINK SWD 下载和调试。

### 7.4 `Core/Src/stm32f1xx_it.c`

`it` 是 interrupt 的缩写。

这个文件放中断服务函数。

当前默认工程里主要有：

- `NMI_Handler`
- `HardFault_Handler`
- `MemManage_Handler`
- `BusFault_Handler`
- `UsageFault_Handler`
- `SVC_Handler`
- `DebugMon_Handler`
- `PendSV_Handler`
- `SysTick_Handler`

最常见的是：

```c
void SysTick_Handler(void)
{
  HAL_IncTick();
}
```

它的作用：

- 每 1ms 增加 HAL 的系统 tick。
- `HAL_Delay()` 等函数依赖它。

当前工程虽然配置了 USART1，但没有打开 USART1 中断，所以这里没有 `USART1_IRQHandler()`。

如果以后在 CubeMX 里打开串口中断，通常会生成类似：

```c
void USART1_IRQHandler(void)
{
  HAL_UART_IRQHandler(&huart1);
}
```

### 7.5 `Core/Inc/stm32f1xx_it.h`

这是 `stm32f1xx_it.c` 的头文件。

它声明中断函数，例如：

```c
void NMI_Handler(void);
void HardFault_Handler(void);
void SysTick_Handler(void);
```

启动文件中的向量表会引用这些函数名。

### 7.6 `Core/Src/system_stm32f1xx.c`

这个文件属于 CMSIS 系统层。

主要提供：

- `SystemInit()`
- `SystemCoreClock`
- `SystemCoreClockUpdate()`

其中：

```c
SystemInit()
```

会在进入 `main()` 之前被启动文件调用。

注意：

```text
SystemInit() 不是 CubeMX 里的 SystemClock_Config()。
```

两者关系：

| 函数 | 调用时机 | 作用 |
| --- | --- | --- |
| `SystemInit()` | 复位后、进入 main 前 | 早期系统初始化、向量表相关 |
| `SystemClock_Config()` | main 里由用户代码调用 | 按 CubeMX 配置设置 RCC 时钟 |

### 7.7 `Core/Inc/stm32f1xx_hal_conf.h`

这是 HAL 配置文件。

它决定哪些 HAL 模块参与编译。

当前启用的关键模块包括：

```c
#define HAL_MODULE_ENABLED
#define HAL_GPIO_MODULE_ENABLED
#define HAL_UART_MODULE_ENABLED
#define HAL_CORTEX_MODULE_ENABLED
#define HAL_DMA_MODULE_ENABLED
#define HAL_FLASH_MODULE_ENABLED
#define HAL_EXTI_MODULE_ENABLED
#define HAL_PWR_MODULE_ENABLED
#define HAL_RCC_MODULE_ENABLED
```

它还定义了时钟常量：

```c
#define HSE_VALUE 8000000U
#define HSI_VALUE 8000000U
#define LSE_VALUE 32768U
```

如果你的外部晶振不是 8 MHz，或者你真的使用了 HSE，就必须保证这里或编译宏里的 `HSE_VALUE` 和实际硬件一致。

新手不要随便改这个文件，优先通过 CubeMX 配置生成。

## 8. `Drivers`：官方库目录

`Drivers` 是 ST 官方库，不是你主要写业务逻辑的地方。

它主要分两类：

```text
Drivers/CMSIS
Drivers/STM32F1xx_HAL_Driver
```

### 8.1 `Drivers/CMSIS`

CMSIS 是 ARM Cortex-M 的标准接口层。

它提供：

- Cortex-M3 内核寄存器定义。
- NVIC、SysTick、SCB 等内核访问接口。
- STM32F1 设备头文件。
- 启动文件模板。
- DSP、NN、RTOS 等可选组件。

本工程最关键的 CMSIS 文件：

| 文件 | 作用 |
| --- | --- |
| `core_cm3.h` | Cortex-M3 内核定义 |
| `stm32f1xx.h` | 根据芯片宏选择具体设备头文件 |
| `stm32f103xb.h` | STM32F103xB 寄存器、外设、中断定义 |
| `system_stm32f1xx.h` | 系统初始化相关声明 |

为什么需要 CMSIS：

```text
没有 CMSIS，代码就不知道 GPIOA、USART1、RCC、NVIC 这些寄存器和中断号怎么定义。
```

### 8.2 `Drivers/STM32F1xx_HAL_Driver`

这是 STM32F1 的 HAL/LL 驱动库。

结构：

```text
Inc/  头文件
Src/  源文件
```

例如：

| 文件 | 作用 |
| --- | --- |
| `stm32f1xx_hal.h` | HAL 总入口头文件 |
| `stm32f1xx_hal_gpio.h/.c` | GPIO HAL 驱动 |
| `stm32f1xx_hal_uart.h/.c` | UART HAL 驱动 |
| `stm32f1xx_hal_rcc.h/.c` | RCC 时钟驱动 |
| `stm32f1xx_hal_cortex.h/.c` | Cortex 相关 HAL |
| `stm32f1xx_hal_flash.h/.c` | Flash 相关 HAL |
| `stm32f1xx_ll_gpio.h/.c` | LL 风格 GPIO 驱动 |

注意：

```text
Drivers 里有很多文件，但当前项目不一定全部编译。
```

当前 `.mxproject` 中记录的已使用 HAL 源文件包括：

- `stm32f1xx_hal_uart.c`
- `stm32f1xx_hal.c`
- `stm32f1xx_hal_rcc.c`
- `stm32f1xx_hal_rcc_ex.c`
- `stm32f1xx_hal_gpio.c`
- `stm32f1xx_hal_gpio_ex.c`
- `stm32f1xx_hal_dma.c`
- `stm32f1xx_hal_cortex.c`
- `stm32f1xx_hal_pwr.c`
- `stm32f1xx_hal_flash.c`
- `stm32f1xx_hal_flash_ex.c`
- `stm32f1xx_hal_exti.c`

这些库文件提供 `HAL_UART_Init()`、`HAL_GPIO_Init()`、`HAL_RCC_OscConfig()` 等函数。

## 9. `EWARM`：IAR 工程目录

`EWARM` 是 IAR Embedded Workbench 的工程目录。

如果你使用 IAR，就打开这里的工程文件。

### 9.1 `EWARM/Project.eww`

`.eww` 是 IAR workspace 文件。

它指向项目文件：

```xml
<path>$WS_DIR$\example.ewp</path>
```

可以理解成：

```text
IAR 工作区入口。
```

### 9.2 `EWARM/example.ewp`

`.ewp` 是 IAR project 文件。

它记录：

- 工程名称。
- 使用的芯片/内核。
- 编译器选项。
- include path。
- 宏定义。
- 源文件列表。
- 汇编文件。
- 链接脚本。
- 输出路径。

它对 IAR 来说类似于：

```text
这个项目该编译哪些文件，怎么编译，怎么链接。
```

当前工程 include path 包括：

```text
../Drivers/STM32F1xx_HAL_Driver/Inc
../Drivers/STM32F1xx_HAL_Driver/Inc/Legacy
../Drivers/CMSIS/Device/ST/STM32F1xx/Include
../Drivers/CMSIS/Include
../Core/Inc
```

宏定义包括：

```text
USE_HAL_DRIVER
STM32F103xB
```

这些宏非常关键：

- `USE_HAL_DRIVER` 让工程使用 HAL。
- `STM32F103xB` 让 `stm32f1xx.h` 选择 `stm32f103xb.h`。

### 9.3 `EWARM/example.ewd`

`.ewd` 是 IAR debug 配置文件。

它记录：

- 调试器设置。
- 是否下载程序。
- 下载后是否校验。
- 是否运行到 `main`。
- C-SPY 调试配置。

它不是业务代码。

### 9.4 `EWARM/startup_stm32f103xb.s`

这是启动文件。

它是整个程序最早执行的代码之一。

它提供：

- 初始栈顶。
- 中断向量表。
- Reset_Handler。
- 弱定义的默认中断处理函数。
- 调用 `SystemInit()`。
- 跳转到 IAR 运行库入口，再进入 `main()`。

向量表里能看到：

```text
Reset_Handler
NMI_Handler
HardFault_Handler
SysTick_Handler
USART1_IRQHandler
USART2_IRQHandler
...
```

这些名字必须和 `stm32f1xx_it.c` 里的中断函数匹配。

如果某个中断没有在 `stm32f1xx_it.c` 中实现，通常会落到启动文件里的默认弱函数。

### 9.5 `EWARM/stm32f103xb_flash.icf`

这是 IAR 链接配置文件，用于“程序烧到 Flash 里运行”的模式。

它定义：

```text
向量表地址：0x08000000
ROM 区域： 0x08000000 到 0x0801FFFF
RAM 区域： 0x20000000 到 0x20004FFF
栈大小：   0x400
堆大小：   0x200
```

它决定：

- `.text` 代码放进 Flash。
- 只读数据放进 Flash。
- 全局变量放进 RAM。
- 栈和堆放进 RAM。
- 向量表放在 `0x08000000`。

重要提醒：

`STM32F103C8T6` 标称 Flash 常见为 64 KB，但这里的 `stm32f103xb_flash.icf` ROM 结束地址是 `0x0801FFFF`，对应 128 KB 区间。

很多 F103C8 小系统板实际芯片或兼容芯片可能能用 128 KB，但严肃项目里应该核对芯片真实 Flash 容量。程序如果超过真实 Flash 容量，会出现烧录失败或运行异常。

### 9.6 `EWARM/stm32f103xb_sram.icf`

这是 IAR 链接配置文件，用于“程序放在 SRAM 中调试运行”的模式。

它定义：

```text
向量表地址：0x20000000
ROM 区域： 0x20000000 到 0x200013FF
RAM 区域： 0x20001400 到 0x20004FFF
```

这不是常规烧录到 Flash 的方式。

新手阶段主要看 `stm32f103xb_flash.icf`。

## 10. `main.c`、`msp.c`、`it.c` 三者怎么配合

这三个文件是理解 CubeMX 工程的关键。

### 10.1 `main.c` 管流程

`main.c` 决定：

- 程序启动后初始化什么。
- 初始化顺序是什么。
- 主循环做什么。

例如：

```c
MX_USART1_UART_Init();
```

### 10.2 `stm32f1xx_hal_msp.c` 管底层资源

当 `main.c` 调用：

```c
HAL_UART_Init(&huart1);
```

HAL 内部会调用：

```c
HAL_UART_MspInit(&huart1);
```

然后 `msp.c` 负责：

- 打开 USART1 时钟。
- 配置 PA9/PA10。
- 必要时配置 DMA。
- 必要时配置 NVIC。

### 10.3 `stm32f1xx_it.c` 管中断入口

如果外设使用中断，中断发生后 CPU 会根据向量表进入对应函数。

例如以后打开 USART1 中断后，流程会变成：

```text
USART1 收到数据
        ↓
NVIC 触发 USART1 中断
        ↓
startup 向量表找到 USART1_IRQHandler
        ↓
进入 stm32f1xx_it.c 的 USART1_IRQHandler
        ↓
调用 HAL_UART_IRQHandler(&huart1)
        ↓
HAL 触发用户回调 HAL_UART_RxCpltCallback
```

当前默认工程没有生成 USART1 中断函数，因为 `.ioc` 里没有打开 USART1 global interrupt。

## 11. 构建时这些文件怎么被编译链接

以 IAR 工程为例，构建流程大致是：

```text
IAR 读取 example.eww
        ↓
IAR 打开 example.ewp
        ↓
根据 .ewp 找到 Core、Drivers、startup、include path、宏定义
        ↓
编译 main.c、msp.c、it.c、system_stm32f1xx.c
        ↓
编译用到的 HAL 源文件
        ↓
汇编 startup_stm32f103xb.s
        ↓
使用 stm32f103xb_flash.icf 链接
        ↓
生成可下载的固件
        ↓
C-SPY 调试器下载到芯片并调试
```

几个关键点：

- `.ioc` 不参与编译，它只给 CubeMX 重新生成代码用。
- `.mxproject` 不参与编译，它给 CubeMX 记录生成状态。
- `Drivers` 里不是所有文件都编译，项目文件决定哪些参与。
- `startup_stm32f103xb.s` 必须参与链接，否则没有复位入口和向量表。
- `.icf` 必须参与链接，否则链接器不知道 Flash/RAM 地址。

## 12. 头文件包含关系

常见包含链路：

```text
main.c
  ↓
main.h
  ↓
stm32f1xx_hal.h
  ↓
stm32f1xx_hal_conf.h
  ↓
按启用模块包含 stm32f1xx_hal_gpio.h / stm32f1xx_hal_uart.h 等
  ↓
stm32f1xx.h
  ↓
stm32f103xb.h
  ↓
core_cm3.h
```

这条链路让你的代码能同时看到：

- HAL 函数。
- 外设结构体。
- 寄存器定义。
- 中断号。
- Cortex-M3 内核定义。

如果 include path 或宏定义错了，就可能出现：

- 找不到 `stm32f1xx_hal.h`。
- 找不到 `stm32f103xb.h`。
- `GPIOA` 未定义。
- `USART1` 未定义。
- `HAL_UART_Init` 未定义。

## 13. `USER CODE BEGIN/END` 是什么

CubeMX 生成的文件里有很多这样的标记：

```c
/* USER CODE BEGIN 2 */

/* USER CODE END 2 */
```

它们的意义是：

```text
CubeMX 重新生成代码时，会尽量保留这些区域里的用户代码。
```

你应该把自己的代码写在这些区域里。

不要随便改 CubeMX 自动生成区域，例如：

```c
huart1.Instance = USART1;
huart1.Init.BaudRate = 115200;
```

如果要改串口波特率，推荐去 CubeMX 里改，然后重新生成。

## 14. 这个默认项目现在能做什么

当前工程配置很少。

它目前具备：

- STM32F103C8Tx 基础启动。
- HAL 初始化。
- SysTick。
- SWD 调试接口。
- USART1，PA9/PA10，115200 8N1。
- 空主循环。

它还没有：

- LED 控制。
- OLED 驱动。
- GPS/北斗接收。
- 串口中断接收。
- 定时器。
- ADC。
- I2C。
- SPI。
- 业务逻辑。

所以它是一个“干净的默认骨架工程”。

## 15. 如果要在这个工程上继续开发，应该改哪里

### 15.1 添加普通业务代码

可以在 `main.c` 的用户区写：

```c
/* USER CODE BEGIN 2 */
// 初始化完成后执行一次
/* USER CODE END 2 */

while (1)
{
  /* USER CODE BEGIN 3 */
  // 循环执行
  /* USER CODE END 3 */
}
```

### 15.2 添加自己的模块

推荐新建：

```text
Core/Inc/app.h
Core/Src/app.c
```

然后在 `main.c` 的用户区包含并调用：

```c
/* USER CODE BEGIN Includes */
#include "app.h"
/* USER CODE END Includes */

/* USER CODE BEGIN 2 */
App_Init();
/* USER CODE END 2 */

while (1)
{
  /* USER CODE BEGIN 3 */
  App_Run();
  /* USER CODE END 3 */
}
```

如果使用 IAR，还要把新的 `.c` 文件加入 `example.ewp` 工程，否则不会参与编译。

如果使用 CubeMX 重新生成，不要指望它自动知道你手动创建的新文件。

### 15.3 添加新外设

推荐流程：

```text
打开 example.ioc
        ↓
配置新外设和引脚
        ↓
生成代码
        ↓
检查 main.c 里是否新增 MX_xxx_Init()
        ↓
检查 msp.c 里是否新增 GPIO/NVIC/DMA 配置
        ↓
在 USER CODE 区写业务逻辑
```

### 15.4 添加中断

如果在 CubeMX 打开外设中断，通常会变化：

- `.ioc` 里 NVIC 记录改变。
- `stm32f1xx_it.c` 增加对应 IRQHandler。
- `stm32f1xx_hal_msp.c` 增加 NVIC 配置。
- `main.c` 可能增加中断接收启动代码，需要你自己写。

## 16. 默认项目和 CMake 项目的区别

这个 `example` 是 EWARM 工程。

你之前的 VS Code 方案更偏向 CMake 工程。

区别：

| 对比 | 当前 example | CMake 工程 |
| --- | --- | --- |
| 工具链目标 | IAR EWARM | CMake + Ninja + Arm GNU 或 ST Bundle |
| 工程文件 | `.eww`、`.ewp`、`.ewd` | `CMakeLists.txt`、`CMakePresets.json` |
| 链接脚本 | `.icf` | `.ld` |
| 启动文件 | EWARM 版本 `.s` | GCC/Clang 版本 `.s` |
| 构建入口 | IAR IDE | CMake / VS Code 插件 |
| 适用场景 | 用 IAR 开发 | 用 VS Code/CMake 开发 |

两者 `Core` 和 `Drivers` 的思想类似，但构建系统不同。

## 17. 新手最应该记住的文件

按重要程度：

| 文件 | 为什么重要 |
| --- | --- |
| `example.ioc` | CubeMX 配置源头 |
| `Core/Src/main.c` | 主程序入口 |
| `Core/Inc/main.h` | 主程序公共定义 |
| `Core/Src/stm32f1xx_hal_msp.c` | 外设底层 GPIO/时钟/NVIC 配置 |
| `Core/Src/stm32f1xx_it.c` | 中断入口 |
| `Core/Inc/stm32f1xx_hal_conf.h` | HAL 模块开关 |
| `EWARM/startup_stm32f103xb.s` | 复位入口和向量表 |
| `EWARM/stm32f103xb_flash.icf` | Flash/RAM 地址和栈堆配置 |
| `EWARM/example.ewp` | IAR 编译工程配置 |

## 18. 不要随便改的文件

不建议新手手动改：

- `.mxproject`
- `Drivers` 下的官方库文件
- `EWARM/example.ewp`
- `EWARM/example.ewd`
- `EWARM/startup_stm32f103xb.s`
- `EWARM/*.icf`
- `Core/Src/system_stm32f1xx.c`

除非你明确知道自己在改什么。

正常开发优先改：

- `.ioc`
- `main.c` 的 USER CODE 区
- 自己新建的 `.c/.h` 模块

## 19. 总结：这些文件的关系

可以用下面这张图理解：

```text
example.ioc
  记录 CubeMX 配置
        ↓ 生成
Core/
  main.c / main.h / msp.c / it.c / system.c
        ↓ 依赖
Drivers/
  CMSIS + HAL
        ↓ 被工程引用
EWARM/example.ewp
  指定编译哪些 Core 和 Drivers 文件
        ↓ 编译/汇编
EWARM/startup_stm32f103xb.s
  提供复位入口和向量表
        ↓ 链接
EWARM/stm32f103xb_flash.icf
  指定 Flash/RAM 布局
        ↓ 生成固件
下载到 STM32 运行
```

最终理解：

```text
.ioc 是配置源头。
Core 是你的工程代码。
Drivers 是官方库。
EWARM 是 IAR 构建和调试配置。
startup 决定程序怎么启动。
icf 决定程序放进哪段 Flash/RAM。
main.c 决定你的程序真正做什么。
```

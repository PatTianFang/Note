# 02-答疑解惑：STM32 硬件连接为什么要这样判断

> 更新时间：2026-05-30。  
> 本文回答硬件连接设计中最容易混淆的几个问题。默认以 `STM32F103C8T6 小系统板` 这类 3.3V STM32 为例，但方法适用于大多数 STM32 项目。  
> 核心思想：不要只问“这根线接哪里”，而要能证明“这样接不会烧、能识别、协议对、软件也配得上”。

## 1. 为什么要先看电源、再看电平、再看协议

这三句话是 STM32 硬件连接的判断顺序：

```text
先看电源，再看信号。
先看电平，再接 IO。
先看协议，再选引脚。
```

它们不是口号，而是为了避免三类最常见的问题：

| 顺序 | 先判断什么 | 防止什么问题 |
| --- | --- | --- |
| 1 | 电源 | 模块不工作、反复重启、烧模块、烧稳压芯片 |
| 2 | 电平 | 5V 信号直接打到 STM32 3.3V IO，导致通信异常或烧 IO |
| 3 | 协议 | 把 UART 当 I2C 接、把 SPI 屏按 I2C 配、软件外设和硬件接线不一致 |

### 1.1 先看电源，再看信号

电源是所有信号的基础。

一个模块只要供电不对，后面的 TX、RX、SCL、SDA、MOSI、MISO 全都没有意义。模块可能完全不工作，也可能工作一会儿就复位，还可能输出异常电平。

接模块前先问：

| 要查的内容 | 怎么看 |
| --- | --- |
| 模块 VCC 接几伏 | 看模块丝印、说明书、数据手册 `Recommended Operating Conditions` |
| 电流需要多大 | 看 `Current Consumption`、`Peak Current`、模块应用说明 |
| STM32 板上的 3.3V 能不能带动 | 看开发板原理图、稳压芯片型号、稳压芯片最大输出电流 |
| 是否需要 5V 和 3.3V 两路电源 | 看模块是否有板载稳压、是否支持宽电压输入 |
| 是否必须共地 | 几乎所有有线通信都必须共地 |

为什么电源要放第一位？

1. 电源电压接错可能直接损坏硬件。比如只支持 3.3V 的传感器接到 5V。
2. 电源电流不够会导致“看起来像软件 bug”的问题。比如 GPS 搜星时电流上升，3.3V 掉压，模块重启，STM32 就一直收不到稳定数据。
3. 没有共地时，信号没有共同参考点。STM32 以自己的 GND 判断高低电平，模块以模块自己的 GND 判断高低电平。如果两边 GND 不连在一起，所谓高电平、低电平就不可靠。

正确做法：

```text
先接 VCC/GND
确认 VCC 电压正确
确认没有短路
确认模块不异常发热
再接信号线
```

### 1.2 先看电平，再接 IO

电源电压和 IO 电平不是同一个概念。

有些模块可以 5V 供电，是因为模块板上有稳压芯片，可以把 5V 变成芯片内部需要的 3.3V。但它的信号线 TX/RX/SDA/SCL 不一定能承受 5V，也不一定输出 3.3V。

接 STM32 IO 前必须确认两件事：

1. 外部模块输出到 STM32 的电压，STM32 能不能承受。
2. STM32 输出到外部模块的电压，模块能不能识别。

例如：

| 情况 | 能不能直接接 |
| --- | --- |
| 3.3V 模块 TX -> STM32 RX | 通常可以 |
| 5V TTL 模块 TX -> STM32 普通 RX | 通常不可以，除非该 STM32 引脚明确 5V 容忍 |
| 5V 模拟传感器输出 -> STM32 ADC | 不可以，ADC 输入不能超过 VDDA |
| RS232 模块输出 -> STM32 UART | 不可以，RS232 不是 TTL 电平，需要 MAX3232 之类转换 |
| RS485 A/B -> STM32 UART | 不可以，RS485 是差分总线，需要 RS485 收发器 |

电平不匹配时，不是改代码能解决的，必须改硬件连接方式。

### 1.3 先看协议，再选引脚

协议决定需要几根线、线的方向、是否需要上拉、是否需要片选、STM32 要打开哪个内部外设。

同样是“传数据”，UART、I2C、SPI 的接法完全不同：

| 协议 | 典型线数 | 方向关系 | STM32 常用外设 |
| --- | --- | --- | --- |
| UART | TX、RX、GND | TX/RX 交叉 | USART/UART |
| I2C | SCL、SDA、GND | 两根线都是总线，通常不交叉 | I2C |
| SPI | SCK、MOSI、MISO、CS、GND | 主从方向明确 | SPI 或 GPIO 模拟 SPI |
| ADC | 模拟信号、GND | 模块输出模拟电压，STM32 输入 | ADC |
| PWM | PWM 信号、GND | STM32 输出脉冲 | TIM |
| GPIO | 一根或多根普通 IO | 看输入/输出定义 | GPIO |

所以不能先随便找几个空闲引脚接上，再想软件怎么配。

正确顺序是：

```text
模块手册说明它用 UART
        -> 选 STM32 的 USARTx_TX / USARTx_RX 引脚
        -> 确认这些引脚在板子上可用
        -> 确认电平兼容
        -> 接线
        -> CubeMX 配 USARTx
```

如果模块说明它用 I2C，就应该去找 STM32 的 I2C_SCL/I2C_SDA 引脚，而不是随便找 TX/RX。

## 2. IO 电平怎么看，为什么决定能不能直接接 STM32

### 2.1 什么是 IO 电平

IO 电平指的是信号线上的逻辑高电平和逻辑低电平范围。

对数字电路来说，信号不是只看“有没有电压”，而是看这个电压会被识别为 `0` 还是 `1`。

以 3.3V 逻辑为例：

```text
低电平：接近 0V，被识别为 0
高电平：接近 3.3V，被识别为 1
```

但实际判断不是简单地等于 0V 或等于 3.3V，而是看芯片手册里的阈值：

| 参数 | 含义 |
| --- | --- |
| `VIH` | 输入被识别为高电平所需的最低电压 |
| `VIL` | 输入被识别为低电平所需的最高电压 |
| `VOH` | 输出高电平时能保证输出到多少电压以上 |
| `VOL` | 输出低电平时能保证输出到多少电压以下 |
| `Absolute Maximum Ratings` | 绝对最大额定值，超过可能损坏芯片 |

判断能不能接，不是只看“这个模块是 3.3V 还是 5V”，而是要看输出端的 `VOH/VOL` 和输入端的 `VIH/VIL/最大耐压` 是否匹配。

### 2.2 怎么看模块的 IO 电平

优先按这个顺序查：

1. 模块数据手册。
2. 模块原理图。
3. 模块卖家说明。
4. 模块芯片 datasheet。
5. 实测信号电压，前提是已经确认不会损坏仪器和芯片。

重点看这些关键词：

```text
IO Voltage
Logic Level
Interface Voltage
TTL Level
CMOS Level
3.3V Compatible
5V Tolerant
Electrical Characteristics
Absolute Maximum Ratings
```

注意几个容易误解的说法：

| 说明 | 真实含义 |
| --- | --- |
| `VCC: 3.3V-5V` | 可能只是供电范围，不代表 IO 一定 5V 安全 |
| `TTL Serial` | 说明是 TTL 串口，不是 RS232，但还要看是 3.3V TTL 还是 5V TTL |
| `5V compatible` | 要看是供电兼容、输入兼容，还是所有 IO 都兼容 |
| `I2C interface` | 要看 SCL/SDA 上拉到几伏 |
| `Analog output 0-5V` | 不能直接进 STM32 3.3V ADC |

### 2.3 怎么看 STM32 的 IO 能不能承受

看 STM32 对应型号的 datasheet，不是只看 Reference Manual。

需要查：

| 要查的项目 | 意义 |
| --- | --- |
| `Pinout and pin description` | 这个引脚是什么功能，是否是普通 IO、ADC、复用功能 |
| `I/O port characteristics` | 输入输出电平、电流能力 |
| `FT` 或 `5V tolerant` 标记 | 该引脚是否 5V 容忍 |
| `ADC characteristics` | ADC 输入范围、采样要求 |
| `Absolute maximum ratings` | 引脚允许的最大电压 |

特别注意：

1. 不是所有 STM32 引脚都 5V 容忍。
2. 5V 容忍通常只针对数字输入，不代表 ADC 模拟输入也能接 5V。
3. 引脚配置成模拟输入时，不要按 5V 容忍理解。
4. STM32 自己输出高电平通常接近 3.3V，不能输出 5V。
5. 即使某个输入能承受 5V，也要确认外部模块能不能识别 STM32 的 3.3V 输出。

### 2.4 为什么 IO 电平决定能不能直接接 STM32

因为 IO 连接本质上是“一个芯片输出，另一个芯片输入”。

判断要同时满足三件事：

```text
输出端的高电平 -> 输入端能识别为高
输出端的低电平 -> 输入端能识别为低
输出端最高电压 -> 不超过输入端最大允许电压
```

例如模块 TX 接 STM32 RX：

| 判断项 | 要满足 |
| --- | --- |
| 模块 TX 输出高电平 | STM32 RX 能识别为高 |
| 模块 TX 输出低电平 | STM32 RX 能识别为低 |
| 模块 TX 最高电压 | 不超过 STM32 RX 允许电压 |

反方向 STM32 TX 接模块 RX：

| 判断项 | 要满足 |
| --- | --- |
| STM32 TX 输出高电平 | 模块 RX 能识别为高 |
| STM32 TX 输出低电平 | 模块 RX 能识别为低 |
| STM32 TX 电压 | 不会损坏模块 RX |

常见结论：

| 模块信号 | STM32 接法 |
| --- | --- |
| 3.3V TTL UART | 通常可直接接 UART |
| 5V TTL UART | 模块 TX 到 STM32 RX 要确认 5V 容忍，否则加电平转换或分压 |
| RS232 | 不能直接接 UART，要 MAX3232 |
| RS485 | 不能直接接 UART，要 RS485 收发器 |
| 0-3.3V 模拟量 | 可进 ADC，但要确认不超过 VDDA |
| 0-5V 模拟量 | 不能直接进 ADC，要分压或运放调理 |
| I2C 上拉到 5V | 对 3.3V STM32 风险很高，通常应上拉到 3.3V 或加电平转换 |

### 2.5 电平不匹配怎么办

常见处理方式：

| 场景 | 处理方法 |
| --- | --- |
| 5V 数字输出到 STM32 3.3V 输入 | 分压、电平转换芯片、三极管/MOS 管转换 |
| STM32 3.3V 输出到 5V 模块输入，模块不认 3.3V 高电平 | 电平转换芯片、晶体管、专用缓冲器 |
| I2C 两边电压不同 | 双向 I2C 电平转换模块，或统一上拉到双方都安全的电压 |
| RS232 和 STM32 UART | MAX3232 |
| RS485 和 STM32 UART | MAX3485、SP3485 等 RS485 收发器 |
| CAN 总线和 STM32 CAN | CAN 收发器，如 TJA1050、SN65HVD230 |
| 模拟量超过 3.3V | 电阻分压、运放缩放、隔离采样 |

最重要的原则：

```text
不确定电平时，先不要接 STM32 IO。
先查资料，必要时用万用表或示波器确认。
```

## 3. 通信方式有哪些，为什么接法完全不同

通信方式不同，差异不只是“名字不同”，而是下面这些都不同：

1. 需要几根线。
2. 每根线方向是什么。
3. 是否需要共用总线。
4. 是否需要上拉。
5. 是否需要片选。
6. 是否需要外部收发器。
7. STM32 要打开哪个外设。
8. 软件要设置哪些参数。
9. 调试时应该看什么现象。

### 3.1 UART

UART 是异步串口，常见于 GPS、北斗、蓝牙、WiFi、串口屏、调试日志。

典型线：

```text
TX
RX
GND
```

接线规则：

```text
模块 TX -> STM32 RX
模块 RX -> STM32 TX
模块 GND -> STM32 GND
```

为什么 TX/RX 要交叉？

TX 是发送，RX 是接收。一个设备发出的数据，必须接到另一个设备的接收端。

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 外设 | USARTx/UARTx |
| 模式 | Asynchronous |
| 波特率 | 必须和模块一致，如 GPS 常见 `9600` |
| 数据位 | 常见 8 bit |
| 校验 | 常见 None |
| 停止位 | 常见 1 bit |
| 流控 | 通常 None，除非模块有 RTS/CTS |
| 中断 | 连续接收数据时建议开启 RX 中断 |
| DMA | 大量连续数据时可用 DMA 接收 |

常见错误：

- TX/RX 没交叉。
- 没有共地。
- 波特率不一致。
- 把 RS232 当 TTL UART 直接接。
- 5V TTL 直接接到不耐 5V 的 STM32 引脚。

### 3.2 I2C

I2C 是两线总线，常见于传感器、EEPROM、RTC、4 针 OLED。

典型线：

```text
SCL 时钟
SDA 数据
GND
VCC
```

接线规则：

```text
模块 SCL -> STM32 I2C_SCL
模块 SDA -> STM32 I2C_SDA
模块 GND -> STM32 GND
SCL/SDA 上拉到合适电压，通常 3.3V
```

I2C 和 UART 最大区别：

| 项目 | UART | I2C |
| --- | --- | --- |
| 线的方向 | TX/RX 一发一收 | SCL/SDA 是总线，不按 TX/RX 交叉 |
| 是否需要地址 | 不需要设备地址 | 每个从设备有 I2C 地址 |
| 是否需要上拉 | 通常不需要 | 必须有上拉 |
| 是否能挂多个设备 | 一对一常见 | 一条总线可挂多个设备，地址不能冲突 |
| 时钟 | 双方按波特率约定 | 主机提供 SCL 时钟 |

为什么 I2C 需要上拉？

I2C 通常是开漏输出。设备只能主动把线拉低，不能主动输出高电平。高电平靠上拉电阻拉回 VCC。

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 外设 | I2Cx |
| 模式 | I2C |
| 速度 | 常见 100 kHz、400 kHz |
| 地址模式 | 7-bit 最常见 |
| GPIO 类型 | 复用开漏，CubeMX 会按 I2C 配 |
| 上拉 | 外部 4.7k 或 10k 常见，模块可能自带 |
| 中断/DMA | 简单读写可不用，大量传输可用 |

常见错误：

- SCL/SDA 没上拉。
- 上拉到了 5V。
- 地址写错，7 位地址和 8 位地址混淆。
- 多个模块地址相同。
- 线太长、电容太大，波形变差。
- 把 SPI OLED 的 `SDA` 误认为 I2C 的 `SDA`。

### 3.3 SPI

SPI 是同步串行接口，常见于 OLED、Flash、SD 卡、无线模块、一些高速传感器。

典型线：

```text
SCK
MOSI
MISO
CS
GND
VCC
```

方向关系：

| 信号 | 方向 |
| --- | --- |
| SCK | 主机 -> 从机 |
| MOSI | 主机 -> 从机 |
| MISO | 从机 -> 主机 |
| CS | 主机 -> 从机，通常低有效 |

接线规则：

```text
STM32 SPI_SCK  -> 模块 SCK
STM32 SPI_MOSI -> 模块 MOSI/DIN/SDA
STM32 SPI_MISO <- 模块 MISO/DOUT
STM32 GPIO     -> 模块 CS
STM32 GND      -> 模块 GND
```

注意：很多 OLED 只需要 STM32 写数据，不需要读数据，所以没有 MISO。

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 外设 | SPIx，或 GPIO 模拟 SPI |
| 模式 | Master |
| 数据方向 | 全双工、半双工、只发送，按模块决定 |
| 数据位 | 常见 8 bit |
| CPOL/CPHA | 必须和模块要求一致，即 SPI Mode 0/1/2/3 |
| NSS/CS | 常用软件控制 CS，用普通 GPIO |
| 分频 | SPI 时钟不能超过模块支持 |
| DMA | 屏幕刷图、Flash 大量读写时可用 |

常见错误：

- MOSI/MISO 接反。
- CS 没接或电平不对。
- CPOL/CPHA 模式不对。
- SPI 时钟太快。
- 模块其实是 I2C 版本，却按 SPI 接。
- 看到 OLED 引脚 `SDA` 就误判为 I2C，实际上 SPI OLED 的 `SDA` 常表示 MOSI。

### 3.4 ADC

ADC 是模拟量采集，不是数字通信协议。

典型线：

```text
模拟输出信号 AO
GND
VCC
```

接线规则：

```text
模块 AO -> STM32 ADC_INx
模块 GND -> STM32 GND
模块 VCC -> 合适电源
```

ADC 和 UART/I2C/SPI 的区别：

| 项目 | 数字通信 | ADC |
| --- | --- | --- |
| 信号含义 | 0/1 编码的数据 | 连续变化的电压 |
| STM32 外设 | UART/I2C/SPI | ADC |
| 关键参数 | 波特率、地址、模式 | 电压范围、采样时间、参考电压 |
| 最大风险 | 协议不匹配 | 电压超过 ADC 量程 |

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 外设 | ADCx |
| 通道 | ADC_INx，必须选有 ADC 功能的引脚 |
| GPIO 模式 | Analog |
| 采样时间 | 信号源阻抗越大，采样时间应越长 |
| 对齐方式 | 右对齐常用 |
| 触发方式 | 软件触发、定时器触发 |
| DMA | 多通道连续采样常用 DMA |

常见错误：

- 0-5V 模拟量直接进 3.3V ADC。
- ADC 引脚没有配置成 Analog。
- 没有共地。
- 分压电阻太大，导致采样不准。
- 信号噪声大但没有滤波。

### 3.5 PWM

PWM 是脉宽调制信号，常用于舵机、蜂鸣器、背光、电机调速、LED 调光。

典型线：

```text
PWM 信号
GND
负载电源
```

接线要分清“信号”和“供电”：

```text
STM32 TIM_CHx -> 模块 PWM/Signal
STM32 GND     -> 模块 GND
负载电源      -> 模块 VCC 或驱动电路
```

PWM 引脚通常不能直接带大电流负载。电机、继电器、大功率 LED、蜂鸣器经常需要三极管、MOSFET 或驱动芯片。

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 外设 | TIMx |
| 通道 | CH1/CH2/CH3/CH4 |
| 模式 | PWM Generation |
| 频率 | 由定时器时钟、预分频、自动重装值决定 |
| 占空比 | 由比较值决定 |
| GPIO | 复用推挽输出 |

常见错误：

- 把 PWM 当电源用。
- 引脚没有选到定时器通道。
- 频率不符合模块要求。
- 舵机需要约 50Hz PWM，却输出了几 kHz。
- 电机没有驱动器和续流保护。

### 3.6 GPIO

GPIO 是普通输入输出，常见于按键、LED、蜂鸣器使能、模块复位、片选、数据/命令选择脚。

常见模式：

| 模式 | 用途 |
| --- | --- |
| Input | 读取按键、模块状态脚、中断脚 |
| Output Push-Pull | 控制 LED、CS、DC、RST、EN |
| Output Open-Drain | 线与逻辑、某些总线或电平转换 |
| Alternate Function | 交给 UART/I2C/SPI/TIM 等外设 |
| Analog | ADC 输入，降低数字输入干扰 |

STM32 设置：

| 配置项 | 说明 |
| --- | --- |
| 输入/输出 | 按外部引脚方向决定 |
| Pull-up/Pull-down | 输入脚没有外部稳定电平时要设置 |
| 输出电平初值 | 片选、复位、使能脚尤其重要 |
| 输出速度 | 普通信号 Low/Medium，快速总线可 High |
| 中断 | 按键、模块中断脚可配置 EXTI |

常见错误：

- 输入脚浮空，读数乱跳。
- 输出默认电平导致模块上电误动作。
- GPIO 直接驱动大电流负载。
- 把需要复用功能的引脚配置成普通 GPIO。

### 3.7 CAN、RS485、USB 这类接口

这些接口不能只看 STM32 内部有没有外设，还要看物理层。

| 接口 | STM32 内部外设 | 外部还需要什么 |
| --- | --- | --- |
| CAN | bxCAN/FDCAN，按型号不同 | CAN 收发器、终端电阻 |
| RS485 | UART | RS485 收发器、方向控制 DE/RE、终端电阻 |
| RS232 | UART | MAX3232 之类电平转换芯片 |
| USB | USB FS/HS | D+/D- 差分线、阻抗、ESD、合适时钟 |

例如 RS485 模块常常不是直接 `A/B` 接 STM32，而是：

```text
STM32 UART_TX -> RS485 收发器 DI
STM32 UART_RX <- RS485 收发器 RO
STM32 GPIO    -> RS485 收发器 DE/RE
RS485 A/B     -> 外部总线
```

## 4. 每个引脚方向怎么看，TX/RX、MOSI/MISO 怎么判断

### 4.1 先记住一个原则

引脚名称通常是站在“这个器件自己”的角度命名的。

`GPS_TX` 表示 GPS 模块发送数据，不是 STM32 发送数据。

所以：

```text
GPS_TX -> STM32_RX
GPS_RX <- STM32_TX
```

如果你看到模块引脚写 `TXD`，通常表示模块的发送端。它应该接到 STM32 的接收端。

### 4.2 UART 的方向

UART 只有发送和接收两类数据线：

| 引脚 | 含义 | 连接方法 |
| --- | --- | --- |
| TX | Transmit，发送 | 接对方 RX |
| RX | Receive，接收 | 接对方 TX |
| GND | 参考地 | 必须共地 |

举例：

| 北斗/GPS 模块 | STM32 |
| --- | --- |
| TX | PA3 / USART2_RX |
| RX | PA2 / USART2_TX |
| GND | GND |

如果只需要接收 GPS 数据，可以先只接：

```text
GPS TX -> STM32 RX
GPS GND -> STM32 GND
GPS VCC -> 电源
```

GPS RX 用于给 GPS 发配置命令，不一定第一步就必须接。

### 4.3 SPI 的方向

SPI 要先判断谁是主机，谁是从机。

通常 STM32 是主机，外部 OLED、Flash、传感器是从机。

| 信号 | 全称 | 方向，假设 STM32 是主机 |
| --- | --- | --- |
| SCK | Serial Clock | STM32 -> 模块 |
| MOSI | Master Out Slave In | STM32 -> 模块 |
| MISO | Master In Slave Out | 模块 -> STM32 |
| CS/NSS | Chip Select | STM32 -> 模块 |

所以：

```text
STM32 SCK  -> 模块 SCK
STM32 MOSI -> 模块 MOSI / DIN / SDA / SDI
STM32 MISO <- 模块 MISO / DOUT / SDO
STM32 GPIO -> 模块 CS / NSS
```

SPI 名字容易混淆：

| 模块丝印 | 可能含义 |
| --- | --- |
| `DIN` | Data In，模块输入，接 STM32 MOSI |
| `DOUT` | Data Out，模块输出，接 STM32 MISO |
| `SDI` | Serial Data In，通常接 MOSI |
| `SDO` | Serial Data Out，通常接 MISO |
| `SDA` | 在 I2C 中是数据线，在 SPI OLED 中常常表示 MOSI |
| `D0` | OLED 上常表示 SCK |
| `D1` | OLED 上常表示 MOSI/SDA |

判断方法：

1. 看模块手册的 `Pin Description`。
2. 看接口章节写的是 `I2C` 还是 `SPI`。
3. 看是否有 `CS/DC/RES`。7 针 OLED 有这些脚时通常是 SPI 屏。
4. 看示例代码的引脚定义。
5. 不确定时不要按丝印猜，先查控制芯片资料。

### 4.4 I2C 的方向

I2C 不按 TX/RX 交叉。

```text
STM32 SCL -> 模块 SCL
STM32 SDA -> 模块 SDA
```

SCL 通常由主机输出时钟。SDA 是双向数据线，主机和从机都可能拉低它。

因此 I2C 的关键不是交叉，而是：

1. SCL/SDA 有没有接对。
2. SCL/SDA 有没有上拉。
3. 上拉电压是否安全。
4. 从设备地址是否正确。

### 4.5 GPIO、RST、EN、INT 的方向

常见控制脚方向如下：

| 模块引脚 | 常见方向 | 说明 |
| --- | --- | --- |
| RST/RESET | STM32 -> 模块 | STM32 拉低或拉高复位模块 |
| EN/ENABLE | STM32 -> 模块 | 使能模块电源或芯片 |
| CS/NSS | STM32 -> 模块 | SPI 片选 |
| DC/A0 | STM32 -> 模块 | OLED 命令/数据选择 |
| INT/IRQ | 模块 -> STM32 | 模块通知 STM32 有事件 |
| BUSY | 模块 -> STM32 | 模块忙状态 |
| KEY/Button | 按键 -> STM32 | 通常配置输入，上拉或下拉 |
| LED | STM32 -> LED | 输出控制，但要注意限流 |

方向判断看手册里 `Input`、`Output`、`I/O`：

| 标记 | 含义 |
| --- | --- |
| `I` | 对模块来说是输入，通常由 STM32 输出 |
| `O` | 对模块来说是输出，通常接 STM32 输入 |
| `I/O` | 双向，按协议或时序变化 |
| `P` | 电源脚 |
| `GND` | 地 |

## 5. 是否需要上拉/下拉，怎么看

### 5.1 上拉/下拉是什么

上拉电阻是把信号线通过电阻连接到高电平。

下拉电阻是把信号线通过电阻连接到低电平。

作用是给信号线一个默认状态，防止引脚悬空。

```text
上拉：默认是 1，没有人拉低时保持高电平
下拉：默认是 0，没有人拉高时保持低电平
```

如果输入脚没有上拉、没有下拉、也没有外部电路稳定驱动，就会浮空。浮空输入可能一会儿读 0，一会儿读 1。

### 5.2 哪些地方常需要上拉

| 场景 | 常见做法 | 原因 |
| --- | --- | --- |
| I2C SCL/SDA | 外部上拉到 3.3V | I2C 开漏，只能拉低，不能主动拉高 |
| 按键输入 | 上拉或下拉 | 按键松开时要有确定状态 |
| BOOT0 | 通常下拉 | 保证默认从 Flash 启动 |
| NRST | 通常上拉 | 保证正常运行时不复位 |
| 模块 EN | 看模块要求，常上拉或由 MCU 控制 | 保证默认启用或默认关闭 |
| 模块 CS | 通常上拉或 MCU 初始拉高 | 防止 SPI 设备上电被误选中 |
| 中断输出 INT | 看是否开漏，开漏则要上拉 | 模块只负责拉低产生中断 |
| 1-Wire 总线 | 上拉 | 总线释放后回到高电平 |

### 5.3 哪些地方常需要下拉

| 场景 | 常见做法 | 原因 |
| --- | --- | --- |
| BOOT0 | 下拉到 GND | 防止上电进入系统 Bootloader |
| MOSFET 栅极 | 下拉 | 防止上电时误导通 |
| 使能脚 EN | 看需求，默认关闭时下拉 | 防止模块上电自动启动 |
| 按键输入 | 如果按键按下接 VCC，则输入脚下拉 | 松开时读 0，按下读 1 |

### 5.4 怎么判断是否需要上拉/下拉

按下面顺序看：

1. 模块手册的引脚说明。
2. 模块手册的典型应用电路。
3. 模块原理图，看板上是否已经有电阻。
4. STM32 开发板原理图，看 BOOT0、NRST、按键是否已处理。
5. 通信协议要求，例如 I2C 必须有上拉。
6. 实测信号，如果输入浮动或波形上升很慢，就要检查上下拉。

手册里的常见提示：

```text
Open-drain output, external pull-up required
Internal pull-up
Pull high to enable
Pull low to reset
Active low
Active high
```

### 5.5 上拉/下拉阻值怎么选

常见经验值：

| 用途 | 常见阻值 |
| --- | --- |
| 普通按键 | 10k |
| BOOT0 默认下拉 | 10k 左右常见 |
| NRST 上拉 | 常见 10k，具体看参考电路 |
| I2C 短线低速 | 4.7k 或 10k |
| I2C 较高速或总线电容较大 | 2.2k 到 4.7k，需看波形和电流 |
| MOSFET 栅极下拉 | 10k 到 100k 常见 |

不要机械套数值。I2C 上拉过大，上升沿慢；上拉过小，器件拉低时电流大。

### 5.6 STM32 内部上拉/下拉够不够

STM32 内部上拉/下拉适合：

- 普通按键。
- 简单状态输入。
- 临时测试。
- 某些低速控制脚。

但很多情况下不建议只依赖内部上拉：

| 场景 | 原因 |
| --- | --- |
| I2C | 内部上拉通常太弱，波形上升慢，建议外部上拉 |
| BOOT0 | 启动状态重要，建议外部电阻保证默认状态 |
| NRST | 复位可靠性重要，按参考电路做 |
| 外部长线输入 | 抗干扰能力不足 |

CubeMX 里常见设置：

| 引脚用途 | GPIO Pull 设置 |
| --- | --- |
| 按键一端接 GND | Pull-up |
| 按键一端接 VCC | Pull-down |
| 外部已有明确上拉/下拉 | No pull 或按电路选择 |
| I2C | 通常由 I2C 外设配置开漏，外部上拉 |
| ADC | No pull，Analog 模式 |

## 6. STM32 引脚从硬件上能不能接这个模块，怎么看

判断一个 STM32 引脚能不能接某个模块，不能只看“这个脚空着”。要同时检查芯片、开发板、电气特性、软件复用功能。

### 6.1 第一步：看芯片 datasheet 的引脚功能

要确认这个引脚有没有你需要的功能。

| 需求 | STM32 引脚必须支持 |
| --- | --- |
| UART | USARTx_TX 或 USARTx_RX |
| I2C | I2Cx_SCL 或 I2Cx_SDA |
| SPI | SPIx_SCK、SPIx_MOSI、SPIx_MISO |
| ADC | ADC_INx |
| PWM | TIMx_CHy |
| 外部中断 | 支持 EXTI，对应 GPIO 通常都可用但线号有限制 |
| 普通控制 | GPIO 输出或输入 |

如果某个引脚没有 ADC 功能，就不能拿它直接读模拟电压。

如果某个引脚没有 TIM 通道，就不能直接输出硬件 PWM，只能软件模拟或换引脚。

### 6.2 第二步：看封装和开发板是否引出

同一个 STM32 型号，不同封装引脚数量不同。datasheet 里有的功能，不代表你手上的小系统板一定引出来了。

要看：

1. 芯片封装。
2. 小系统板排针丝印。
3. 开发板原理图。
4. 这个引脚是否真的连到排针。

有些引脚在芯片上存在，但板子没引出，实际就不能接杜邦线使用。

### 6.3 第三步：看是否被板载电路占用

开发板上可能已经占用了某些引脚：

| 引脚类型 | 可能用途 |
| --- | --- |
| PA13/PA14 | SWD 调试 |
| NRST | 复位 |
| BOOT0 | 启动模式 |
| OSC_IN/OSC_OUT | 外部高速晶振 |
| PC14/PC15 | 低速晶振 |
| 某些 GPIO | 板载 LED、按键、蜂鸣器 |
| USB D+/D- | USB |

被占用不是一定不能用，而是要知道后果。

例如占用 PA13/PA14 会影响 ST-LINK 下载调试。初学阶段不建议占用。

### 6.4 第四步：看电气特性是否允许

要确认：

| 检查项 | 原因 |
| --- | --- |
| 引脚是否 5V 容忍 | 外部模块可能输出 5V |
| 是否 ADC 引脚 | 模拟量只能进 ADC 引脚 |
| ADC 输入电压是否超过 VDDA | 超过会损坏或读数错误 |
| IO 输出电流是否够 | 不能直接驱动大电流负载 |
| 引脚速度是否满足 | 高速 SPI、时钟信号对波形要求更高 |
| 上电默认状态是否安全 | 某些引脚复位后会浮空或有特殊状态 |

STM32 IO 通常只能输出很小电流，适合输出控制信号，不适合直接驱动电机、继电器、大功率 LED。

### 6.5 第五步：看复用功能是否冲突

STM32 一个引脚可能有多个复用功能，但同一时间通常只能选一个。

例如一个引脚可能同时支持：

```text
GPIO
USART
TIM
SPI
```

如果你把它配置成 `USART_TX`，它就不是普通 GPIO 输出了。

如果你把它配置成 `TIM_CH1` 输出 PWM，就不能同时当作 I2C_SCL 使用。

CubeMX 的作用就是帮助检查这种冲突。

### 6.6 实际判断流程

可以按这个清单判断：

```text
1. 模块需要什么协议？
2. 这个协议需要哪些 STM32 外设？
3. datasheet 里哪些引脚支持这些外设？
4. 这些引脚在我的板子上有没有引出？
5. 有没有被 SWD、晶振、BOOT、板载 LED 等占用？
6. 电平是否兼容？
7. 电流和驱动能力是否满足？
8. CubeMX 里配置后有没有冲突？
9. 连接表和代码定义是否一致？
```

只要其中一项不满足，就不要直接接。

## 7. UART、I2C、SPI、ADC、TIM 等外设功能是什么，怎么决定连接

这里的“外设”指 STM32 芯片内部的硬件功能模块，不是外面接的传感器或 OLED。

例如：

```text
USART 是 STM32 内部外设。
GPS 模块是外部模块。
GPS 模块通过 UART 协议连接到 STM32 的 USART 外设。
```

### 7.1 UART/USART

功能：

- 串行收发字节。
- 常用于 GPS、蓝牙、WiFi、串口屏、调试日志。
- 适合一对一、低到中等速度的数据传输。

连接怎么决定：

1. 模块手册写 `UART`、`TTL Serial`、`TXD/RXD`。
2. STM32 选择一个 USART/UART 外设。
3. TX/RX 交叉连接。
4. 软件配置波特率、数据位、校验位、停止位。

适合本项目：

```text
北斗/GPS 模块输出 NMEA 文本 -> 使用 STM32 USART2 接收
GPS TX -> PA3 / USART2_RX
GPS RX -> PA2 / USART2_TX
```

### 7.2 I2C

功能：

- 两根线挂多个从设备。
- 常用于低速传感器、RTC、EEPROM、4 针 OLED。
- 通过设备地址区分不同模块。

连接怎么决定：

1. 模块手册写 `I2C`、`SCL/SDA`。
2. STM32 选择 I2C_SCL/I2C_SDA 引脚。
3. SCL 接 SCL，SDA 接 SDA。
4. 检查上拉电阻和上拉电压。
5. 软件配置总线速度和从设备地址。

什么时候不适合：

- 线很长、干扰大。
- 多个模块地址冲突。
- 需要很高带宽。

### 7.3 SPI

功能：

- 主从同步高速通信。
- 常用于 OLED、Flash、SD 卡、无线模块、高速 ADC/DAC。
- 速度比 I2C 通常更高，但占线更多。

连接怎么决定：

1. 模块手册写 `SPI`，并给出 SCK/MOSI/MISO/CS。
2. STM32 选择 SPIx 引脚，或低速场景用 GPIO 模拟。
3. SCK/MOSI/MISO 按主从方向连接。
4. 每个从设备单独一个 CS。
5. 软件配置 SPI 模式、速度、数据位。

本项目 OLED 示例：

```text
7 针 SPI OLED 可以用 GPIO 模拟 SPI
PB8  -> OLED D0/SCK
PB9  -> OLED D1/SDA/MOSI
PB12 -> OLED DC
PB13 -> OLED CS
```

### 7.4 ADC

功能：

- 把模拟电压转换成数字量。
- 用于电位器、模拟传感器、电池电压检测、光敏电阻等。

连接怎么决定：

1. 模块输出是模拟电压，不是数字协议。
2. 选择 STM32 ADC_INx 引脚。
3. 确认电压范围在 `0 到 VDDA` 之间。
4. 必要时分压、滤波、缓冲。
5. 软件配置 ADC 通道、采样时间、触发方式。

注意：

```text
ADC 不能接 UART/I2C/SPI 的数字数据线来“读数据”。
ADC 只能读电压。
```

### 7.5 TIM

TIM 是定时器外设，功能很多，不只是延时。

常见用途：

| 功能 | 用途 |
| --- | --- |
| PWM 输出 | 舵机、LED 调光、电机调速、蜂鸣器 |
| 输入捕获 | 测频率、测脉宽、超声波回波 |
| 输出比较 | 定时翻转引脚、产生精确定时事件 |
| 编码器接口 | 读取旋转编码器 |
| 基本定时 | 周期中断、系统任务调度 |

连接怎么决定：

1. 模块需要 PWM，就找 `TIMx_CHy` 引脚。
2. 模块输出脉冲需要测量，就找支持输入捕获的 TIM 通道。
3. 需要精确定时触发 ADC，就用 TIM 触发 ADC。
4. 软件配置预分频、计数周期、通道模式。

例如舵机：

```text
STM32 TIMx_CHy -> 舵机信号线
舵机 VCC      -> 独立 5V 或合适电源
舵机 GND      -> STM32 GND 共地
```

舵机电源不能直接从 STM32 IO 取。

### 7.6 GPIO

功能：

- 作为普通输入读取高低电平。
- 作为普通输出控制高低电平。
- 作为复用功能交给内部外设。

连接怎么决定：

| 模块需求 | GPIO 配置 |
| --- | --- |
| LED 指示灯 | 输出，注意限流 |
| 按键 | 输入，上拉或下拉 |
| 模块 RST | 输出，按模块要求拉高/拉低 |
| 模块 EN | 输出，控制使能 |
| 模块 INT | 输入或外部中断 EXTI |
| SPI CS | 输出，通常默认拉高 |

GPIO 很灵活，但不能替代所有外设。比如 GPS 持续输出串口数据，优先用硬件 USART，不建议一开始就用 GPIO 软件模拟。

### 7.7 怎么根据模块决定用哪个 STM32 外设

看模块手册，而不是先看 STM32。

| 模块手册写什么 | STM32 应该用什么 |
| --- | --- |
| `UART`, `TXD`, `RXD`, `baudrate` | USART/UART |
| `I2C`, `SCL`, `SDA`, `address` | I2C |
| `SPI`, `SCK`, `MOSI`, `MISO`, `CS`, `CPOL/CPHA` | SPI |
| `Analog Output`, `AO`, `0-3.3V` | ADC |
| `PWM input`, `servo signal` | TIM PWM 输出 |
| `Pulse output`, `frequency output` | TIM 输入捕获或 EXTI |
| `KEY`, `LED`, `EN`, `RST`, `CS`, `DC` | GPIO |
| `RS485 A/B` | UART + RS485 收发器 |
| `CANH/CANL` | CAN 外设 + CAN 收发器 |

判断顺序：

```text
模块协议 -> STM32 内部外设 -> 支持该外设的引脚 -> 电平检查 -> 接线 -> CubeMX 配置
```

## 8. Reference Manual 里的总线、寄存器、时钟树、中断、DMA、Boot 模式分别有什么用

STM32 资料通常分两类：

| 资料 | 主要用途 |
| --- | --- |
| Datasheet | 看芯片外部特性：引脚、电气参数、封装、5V 容忍、ADC 引脚 |
| Reference Manual | 看芯片内部怎么工作：外设、寄存器、总线、时钟、中断、DMA、启动模式 |

做硬件连接时，datasheet 解决“这个脚能不能接”，Reference Manual 解决“这个外设怎么配置、怎么运行”。

### 8.1 外设挂在哪条总线

STM32 内部有多条总线，常见名字包括：

```text
AHB
APB1
APB2
```

不同外设挂在不同总线上。

这有什么用？

1. 决定外设时钟从哪里来。
2. 决定要使能哪个 RCC 时钟。
3. 决定外设最高时钟频率。
4. 影响 UART 波特率、SPI 速度、定时器频率等计算。

以 STM32F103 常见情况举例：

| 外设 | 常见总线 |
| --- | --- |
| GPIO | APB2 |
| AFIO | APB2 |
| USART1 | APB2 |
| USART2/USART3 | APB1 |
| SPI1 | APB2 |
| SPI2 | APB1 |
| I2C1/I2C2 | APB1 |
| TIM1 | APB2 |
| TIM2/TIM3/TIM4 | APB1 |
| ADC1/ADC2 | APB2 |
| DMA | AHB |

实际项目以对应型号 Reference Manual 为准。

为什么你需要知道？

如果 USART2 挂在 APB1，而你的 APB1 时钟配置错了，USART2 波特率就可能不准。GPS 模块明明是 9600，STM32 实际发出来不是 9600，就会乱码或收不到。

### 8.2 寄存器说明

寄存器是控制 STM32 硬件外设的“开关”和“状态表”。

HAL 库、LL 库、CubeMX 最终都是在配置寄存器。

寄存器能做什么？

| 寄存器类型 | 用途 |
| --- | --- |
| 控制寄存器 | 开启外设、设置模式、开启中断 |
| 状态寄存器 | 查看是否接收完成、发送完成、溢出、错误 |
| 数据寄存器 | 收发数据 |
| 配置寄存器 | 设置波特率、分频、极性、采样时间 |

你不一定一开始就直接写寄存器，但要知道寄存器说明有什么用：

1. 当 HAL 函数不工作时，可以看状态位定位原因。
2. 看懂 `RXNE`、`TXE`、`TC`、`OVR`、`BUSY` 等标志。
3. 判断是不是忘了清中断标志。
4. 判断外设是否真的启动。
5. 理解 CubeMX 配置背后改了什么。

例如 UART 常见状态：

| 标志 | 含义 |
| --- | --- |
| RXNE | 接收数据寄存器非空，说明收到字节 |
| TXE | 发送数据寄存器空，可以写入新数据 |
| TC | 发送完成 |
| ORE | 接收溢出，数据来得太快或处理太慢 |

### 8.3 时钟树

时钟树描述 STM32 的时钟从哪里来，又如何分配给 CPU 和外设。

常见时钟源：

| 时钟源 | 含义 |
| --- | --- |
| HSI | 内部高速 RC 时钟 |
| HSE | 外部高速晶振 |
| LSI | 内部低速时钟 |
| LSE | 外部低速 32.768kHz 晶振 |
| PLL | 锁相环，用于倍频 |

常见时钟：

| 时钟 | 用途 |
| --- | --- |
| SYSCLK | 系统主时钟 |
| HCLK | AHB 总线和 CPU 时钟 |
| PCLK1 | APB1 外设时钟 |
| PCLK2 | APB2 外设时钟 |
| ADC Clock | ADC 采样时钟 |
| TIM Clock | 定时器计数时钟 |

时钟树为什么重要？

| 外设 | 时钟错误的后果 |
| --- | --- |
| UART | 波特率不准，串口乱码 |
| I2C | 速度不对，通信失败 |
| SPI | 时钟过快，模块读不稳 |
| TIM/PWM | PWM 频率不对，舵机乱动 |
| ADC | 采样时间不合适，数据不准 |
| USB | 时钟精度不够，USB 不识别 |

CubeMX 里的 Clock Configuration 就是在配置时钟树。

初学阶段建议：

```text
先用简单可靠的时钟配置跑通功能。
确认硬件连接正常后，再优化到更高主频。
```

不要一开始就随便把系统时钟改到 72MHz，却没有确认外部晶振是否真的存在。

### 8.4 中断

中断是外设或引脚主动打断 CPU，让 CPU 及时处理事件。

不用中断时，CPU 需要一直轮询：

```text
有没有收到串口数据？
有没有按键按下？
ADC 转换完了吗？
```

用中断时，事件来了再处理：

```text
串口收到一个字节 -> 进入 UART 中断
按键边沿触发 -> 进入 EXTI 中断
定时器到点 -> 进入 TIM 中断
DMA 搬运完成 -> 进入 DMA 中断
```

常见中断用途：

| 中断 | 用途 |
| --- | --- |
| UART RX 中断 | 接收 GPS、蓝牙、WiFi 数据 |
| EXTI 中断 | 按键、模块 INT/IRQ |
| TIM 中断 | 周期任务、计时 |
| ADC 中断 | 转换完成 |
| DMA 中断 | 数据搬运完成或半完成 |

配置中断通常要做两件事：

1. 外设内部允许产生中断。
2. NVIC 里启用对应中断，并设置优先级。

只做其中一个通常不够。

例如 GPS 持续输出 NMEA 数据，如果不用中断或 DMA，只在主循环偶尔读串口，可能丢数据。

### 8.5 DMA

DMA 是 Direct Memory Access，直接内存访问。

作用是让数据在外设和内存之间搬运，不需要 CPU 一个字节一个字节处理。

常见场景：

| 场景 | DMA 作用 |
| --- | --- |
| UART 连续接收 | 把串口数据自动放进缓冲区 |
| ADC 多通道采样 | 自动把采样结果放进数组 |
| SPI 刷屏 | 自动把显存数据发给 OLED/TFT |
| I2C/SPI 大量读写 | 减少 CPU 占用 |

不用 DMA：

```text
每来一个字节，CPU 处理一次。
```

用 DMA：

```text
外设收到数据 -> DMA 自动写入内存缓冲区 -> 满了或半满再通知 CPU。
```

DMA 不是必须的。

简单项目可以先不用 DMA，先用阻塞函数或中断跑通。数据量大、实时性要求高、CPU 忙不过来时再用 DMA。

DMA 使用时要注意：

1. DMA 通道必须和外设匹配。
2. 缓冲区不能太小。
3. 循环模式和普通模式要选对。
4. 半传输、传输完成中断要处理清楚。
5. 缓冲区数据边界要管理好。

### 8.6 Boot 模式

Boot 模式决定 STM32 上电或复位后从哪里启动程序。

常见启动来源：

| 启动来源 | 用途 |
| --- | --- |
| Main Flash | 正常运行用户程序 |
| System Memory | 进入芯片内置 Bootloader，可用串口/USB 等方式下载，具体看型号 |
| SRAM | 调试或特殊用途 |

BOOT0、BOOT1 或选项字节会影响启动模式。不同 STM32 系列细节不同，要看对应手册。

对初学者最重要的结论：

```text
正常运行程序时，BOOT0 通常要保持为 0。
BOOT0 不能悬空。
```

如果 BOOT0 悬空或被拉高，STM32 可能不从用户 Flash 启动。表现就是：

- 程序明明烧录成功，但复位后不运行。
- LED 不闪。
- OLED 没反应。
- 调试时看起来像程序没进 main。

所以开发板通常会给 BOOT0 加下拉电阻，或者用跳帽选择。

不要随便把 BOOT0 当普通 IO 用。即使某些型号允许复用，也不适合初学阶段使用。

## 9. 实际接线前应该怎么一步步判断

每接一个新模块，都可以按下面模板做。

### 9.1 模块信息表

| 项目 | 要填写的内容 |
| --- | --- |
| 模块名称 | 例如 GPS、OLED、温湿度传感器 |
| 功能 | 定位、显示、采集、通信 |
| 供电电压 | 3.3V、5V、宽电压 |
| 峰值电流 | 电源能不能带动 |
| IO 电平 | 3.3V、5V、RS232、RS485、模拟量 |
| 通信方式 | UART/I2C/SPI/ADC/PWM/GPIO |
| 引脚方向 | 输入、输出、双向 |
| 是否上拉/下拉 | I2C、按键、BOOT、RST、INT 等 |
| 是否需要复位/使能 | RST、EN、CS、DC |
| 初始化参数 | 波特率、I2C 地址、SPI 模式、PWM 频率 |

### 9.2 连接判断表

| 判断问题 | 合格标准 |
| --- | --- |
| VCC 接几伏 | 和模块手册一致 |
| GND 是否共地 | STM32 和模块 GND 连通 |
| 模块输出电平 | 不超过 STM32 输入允许范围 |
| STM32 输出电平 | 模块能识别 |
| 协议是否匹配 | UART 对 USART，I2C 对 I2C，SPI 对 SPI |
| 引脚方向是否正确 | TX 到 RX，MOSI 到从机输入，MISO 到主机输入 |
| 是否需要上拉 | I2C、开漏、中断脚、按键已处理 |
| STM32 引脚是否支持 | datasheet 和 CubeMX 均确认 |
| 是否和调试/启动冲突 | 不占 PA13/PA14、NRST、BOOT0 等关键脚 |
| 软件配置是否一致 | CubeMX、代码宏定义、实际接线一致 |

### 9.3 推荐验证顺序

```text
1. 只接电源和地，测 VCC 是否正确。
2. 断电后测 VCC 与 GND 是否短路。
3. 单独验证模块是否有基本反应。
4. 接通信线，但一次只接一个模块。
5. 用最简单程序验证通信。
6. 再接下一个模块。
7. 最后整机联调。
```

本项目可以这样验证：

```text
STM32 + ST-LINK
        -> 能下载 LED 闪烁
GPS + USB-TTL
        -> 串口助手能看到 NMEA
STM32 + GPS
        -> USART2 能收到数据
STM32 + OLED
        -> OLED 能显示固定文本
STM32 + GPS + OLED
        -> OLED 显示定位状态
```

## 10. 最后总结

硬件连接不是背接线表，而是按证据判断。

最重要的判断链条是：

```text
模块需要什么电源
        -> 模块 IO 电平是多少
        -> 模块用什么通信协议
        -> 每个引脚方向是什么
        -> 是否需要上拉/下拉
        -> STM32 哪些引脚支持这个功能
        -> 是否和调试、启动、晶振、板载电路冲突
        -> CubeMX 和代码是否按同一套接线配置
        -> 单模块验证是否通过
```

记住一句话：

```text
能直接接 STM32 的前提不是“线能插上”，而是电源正确、电平安全、协议匹配、方向正确、引脚支持、软件一致。
```

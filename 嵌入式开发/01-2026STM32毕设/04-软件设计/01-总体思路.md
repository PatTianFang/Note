# 01-总体思路：北斗显示屏 STM32 连接项目的软件设计怎么理解

> 更新时间：2026-05-30。  
> 本文分析对象是：`00-张立超毕设/04-各模块连接的烧录代码/01-北斗显示屏STM32连接/VSCode`。  
> 重点不是只说明“代码能运行”，而是解释这个项目为什么这样设计、软件和硬件怎样对应、每个模块为什么这样选、这样做的好处是什么、后续还能怎么扩展。

## 1. 先给这个项目下一个准确的定义

这个工程不是单纯点亮 OLED，也不是单纯读取 GPS 串口。

它真正完成的是一个小型嵌入式系统闭环：

```text
北斗/GPS 模块接收卫星信号
        -> 模块通过 UART 输出 NMEA 文本数据
        -> STM32 通过 USART2 中断接收字节流
        -> 软件从字节流里捕获完整 RMC 语句
        -> 软件解析 UTC 时间、定位状态、经纬度
        -> 软件把 UTC 转北京时间，把 NMEA 坐标转十进制度
        -> OLED 显示定位状态、时间、经纬度和作者信息
```

所以这个项目的核心价值在于：

1. 硬件上把主控、定位模块、显示模块、下载调试接口连接起来。
2. 软件上把“数据接收、协议解析、数据显示、硬件驱动”分开。
3. 功能上形成“采集 -> 处理 -> 显示”的完整链路。
4. 工程上能烧录、能运行、能定位问题、能继续扩展。

可以把它理解成一个最小版的“定位信息显示终端”。

## 2. 当前工程的真实实现依据

本文以当前源码和 README 的实际实现为准。

实际工程中，关键文件如下：

| 文件 | 作用 |
| --- | --- |
| `Core/Src/main.c` | 系统初始化、USART2 接收启动、主循环调度、UART 回调 |
| `Core/Src/bds_nmea.c` | NMEA 字节流捕获、RMC 解析、坐标和时间格式化 |
| `Core/Src/bds_display.c` | 根据定位结果组织 OLED 显示内容 |
| `Core/Src/oled_spi.c` | GPIO 模拟 SPI，驱动 SSD1306 类 OLED |
| `Core/Src/stm32f1xx_hal_msp.c` | USART2 的 PA2/PA3 GPIO 和 NVIC 初始化 |
| `Core/Src/stm32f1xx_it.c` | USART2 中断入口 |
| `Core/Inc/*.h` | 各模块对外接口 |
| `README.md` | 工程目标、接线、验证和排查说明 |
| `VSCode.ioc` | CubeMX 配置文件，但当前与源码存在部分不一致 |

需要特别注意一个点：

```text
VSCode.ioc 里仍残留 USART1 / PA9 / PA10 配置。
但当前源码和 README 实际使用的是 USART2 / PA2 / PA3。
```

也就是说，理解当前固件时应以这些事实为准：

```text
GPS/BDS 模块：PA2/PA3，USART2，9600 8N1
OLED：PB8/PB9/PB12/PB13，GPIO 模拟 SPI
调试：PA13/PA14，SWD
系统时钟：HSI 8MHz，无 PLL
```

后续如果重新打开 CubeMX 生成代码，需要小心 `.ioc` 覆盖当前 USART2 手写配置。

## 3. 总体硬件和软件搭配关系

整个系统可以分成 4 个硬件角色：

| 硬件 | 在系统中的角色 | 软件对应模块 |
| --- | --- | --- |
| STM32F103C8T6 | 主控，负责接收、解析、显示控制 | `main.c`、HAL、各业务模块 |
| 北斗/GPS 模块 | 定位数据来源，输出 NMEA 文本 | `bds_nmea.c/h` |
| 7 针 SPI OLED | 人机显示界面 | `bds_display.c/h`、`oled_spi.c/h` |
| ST-LINK | 下载和调试 | SWD、CubeProgrammer、调试器 |

硬件和软件的对应关系如下：

```text
北斗/GPS 模块
  TX  -> PA3 / USART2_RX -> UART 中断 -> BDS_NMEA_PushByte()
  RX  <- PA2 / USART2_TX -> 后续可用于配置模块

OLED
  D0/SCK <- PB8  -> OLED_WriteByte() 模拟时钟
  D1/SDA <- PB9  -> OLED_WriteByte() 模拟数据
  DC     <- PB12 -> OLED_WriteCommand()/OLED_WriteData()
  CS     <- PB13 -> 每个字节发送时片选
  RES    <- 板载 R/RESET -> 软件不单独控制，初始化前延时等待

ST-LINK
  SWDIO -> PA13
  SWCLK -> PA14
  GND   -> GND
```

这套组合的特点是：

1. 定位模块用标准 UART，接收简单、稳定。
2. 显示模块用 OLED，显示信息直观，所需引脚不多。
3. OLED 用 GPIO 模拟 SPI，避开硬件 SPI 引脚限制。
4. 主控用 STM32F103C8T6，资源足够，学习资料多，成本低。
5. 软件按模块分层，便于调试和扩展。

## 4. 为什么选 STM32F103C8T6 做主控

STM32F103C8T6 对这个项目是合适的，原因不是它性能最高，而是它刚好满足需求。

### 4.1 资源够用

这个项目需要的资源包括：

| 需求 | STM32F103C8T6 是否满足 |
| --- | --- |
| 一个 UART 接 GPS/BDS | 满足，当前用 USART2 |
| 若干 GPIO 接 OLED | 满足，当前用 PB8/PB9/PB12/PB13 |
| SWD 下载调试 | 满足，PA13/PA14 |
| 解析 NMEA 字符串 | CPU 和 RAM 足够 |
| OLED 文本显示 | Flash 和 RAM 足够 |
| 简单裸机主循环 | 不需要 RTOS |

工程 README 中记录过当前固件资源占用：

```text
FLASH: 14940 B / 64 KB
RAM:   2264 B / 20 KB
```

这说明当前功能对 F103C8T6 来说压力不大。

### 4.2 外设刚好匹配

GPS/BDS 模块输出的是串口文本，STM32F103 有硬件 USART。

OLED 使用 SPI 风格时序，STM32 可以用硬件 SPI，也可以用 GPIO 模拟。当前接线固定在 PB8/PB9/PB12/PB13，用 GPIO 模拟更符合现有硬件。

### 4.3 学习和毕业设计友好

STM32F103C8T6 的优势：

- 资料多。
- 开发板便宜。
- ST-LINK 下载方便。
- HAL、CubeMX、CMake 工程都能支持。
- UART、GPIO、中断这些基础知识都能覆盖。
- 项目复杂度适中，适合作为毕业设计中的功能模块。

这个项目不是为了追求最高性能，而是为了把“硬件连接 + 外设配置 + 协议解析 + 显示输出”讲清楚。

## 5. 为什么北斗/GPS 模块用 UART/USART2

### 5.1 GPS/BDS 模块天然适合 UART

常见北斗/GPS 模块会持续输出 NMEA 文本，例如：

```text
$GNRMC,hhmmss.sss,A,ddmm.mmmm,N,dddmm.mmmm,E,...
```

这种数据本质是 ASCII 字符流，非常适合 UART。

UART 的好处：

| 优点 | 对本项目的意义 |
| --- | --- |
| 接线少 | TX、RX、GND 即可 |
| 模块默认支持 | 大多数 GPS/BDS 模块都默认输出 NMEA UART |
| 调试方便 | 可以先用 USB-TTL 接电脑串口助手验证 |
| 数据是文本 | 解析直观，便于毕业设计说明 |
| STM32 硬件支持 | USART 中断接收可靠 |

### 5.2 为什么用 USART2 而不是软件模拟串口

GPS/BDS 模块会周期性输出数据，常见 1Hz，也可能更高。

如果用软件模拟串口，需要靠 GPIO 和延时精确采样。这样容易受中断、主循环阻塞、OLED 刷新影响。

硬件 USART 的优势：

1. 波特率由硬件产生，稳定。
2. 接收采样由硬件完成，CPU 不需要卡时序。
3. 收到字节后触发中断，软件只处理结果。
4. 后续可以扩展 DMA 接收。

所以当前设计用 USART2 是合理的。

### 5.3 为什么选择 PA2/PA3

当前实际接线是：

| STM32 | GPS/BDS 模块 | 说明 |
| --- | --- | --- |
| `PA2 / USART2_TX` | `RX` | STM32 给模块发命令，当前主要保留扩展 |
| `PA3 / USART2_RX` | `TX` | GPS/BDS 模块输出 NMEA，STM32 接收 |

设计好处：

- PA2/PA3 是 STM32F103 上常用的 USART2 引脚。
- 不占用 PA13/PA14 的 SWD 调试口。
- 可以把 USART1 预留给调试日志或 USB-TTL。
- 和 README 中记录的当前接线一致。

### 5.4 为什么参数是 9600 8N1

当前 `main.c` 中配置：

```text
BaudRate: 9600
WordLength: 8 bit
Parity: None
StopBits: 1
FlowControl: None
```

这就是常说的 `9600 8N1`。

选择原因：

1. 很多 GPS/BDS 模块默认 NMEA 输出就是 9600 8N1。
2. 数据量不大，9600 足够传输 RMC、GGA、GSV 等语句。
3. 配置简单，串口助手也容易验证。
4. 不需要 RTS/CTS 硬件流控，减少接线。

## 6. 为什么 OLED 使用 GPIO 模拟 SPI

### 6.1 当前 OLED 是 7 针 SPI OLED

当前 OLED 接线：

| STM32 | OLED | 含义 |
| --- | --- | --- |
| `GND` | `GND` | 地 |
| `3.3V` | `VDD` | 电源 |
| `PB8` | `D0 / SCK` | SPI 时钟 |
| `PB9` | `D1 / SDA` | SPI 数据，实际相当于 MOSI |
| `R / RESET` | `RES` | 复位 |
| `PB12` | `DC` | 命令/数据选择 |
| `PB13` | `CS` | 片选 |

它不是 4 针 I2C OLED。

如果是 I2C OLED，常见引脚是：

```text
GND / VCC / SCL / SDA
```

而当前 7 针 OLED 有 `D0/SCK`、`D1/SDA`、`DC`、`CS`、`RES`，这就是 SPI 风格控制。

### 6.2 为什么不用硬件 SPI

STM32F103 的硬件 SPI 引脚通常有固定组合，例如 SPI1 常见在 PA5/PA6/PA7。

当前 OLED 已经接在：

```text
PB8  -> SCK
PB9  -> SDA/MOSI
PB12 -> DC
PB13 -> CS
```

这些并不是默认硬件 SPI1 的常规引脚组合。

如果强行改成硬件 SPI，有两个选择：

1. 改硬件接线，把 OLED 挪到硬件 SPI 引脚。
2. 查复用映射和重映射，重新配置 CubeMX 和代码。

当前项目为了快速稳定运行，选择 GPIO 模拟 SPI。

这样做的好处：

| 好处 | 说明 |
| --- | --- |
| 符合现有接线 | 不需要重新插线 |
| 代码可控 | SCK、SDA、DC、CS 全部由软件明确控制 |
| 速度足够 | 128x64 OLED 显示定位信息，不需要高速刷新 |
| 降低 CubeMX 依赖 | 不需要为了 OLED 重新配置硬件 SPI |
| 易理解 | 初学者能看懂每一根线什么时候变高、变低 |

代价是：

| 代价 | 影响 |
| --- | --- |
| CPU 参与每一位发送 | 刷屏时 CPU 被占用 |
| 速度低于硬件 SPI | 不适合大屏高速刷新 |
| 时序靠软件延时 | 需要保证延时足够稳定 |

对当前项目来说，这些代价可以接受。因为 OLED 只显示几行文字，GPS/BDS 数据通常 1 秒更新一次，显示刷新压力很小。

### 6.3 OLED 驱动为什么自包含 GPIO 初始化

`main.c` 只调用：

```c
BDS_Display_Init();
```

显示层再调用：

```c
OLED_Init();
```

而 `OLED_Init()` 内部自己完成：

1. 打开 GPIOB 时钟。
2. 配置 PB8/PB9/PB12/PB13 为推挽输出。
3. 设置 CS、SCK、SDA、DC 初始电平。
4. 等待 OLED 复位稳定。
5. 发送 SSD1306 初始化命令。
6. 清屏。

这样做的好处：

- OLED 底层驱动不依赖 CubeMX 是否配置了这些 GPIO。
- `main.c` 不需要知道 OLED 具体用了哪些脚。
- 如果后面换 OLED 引脚，只改 `oled_spi.h` 和 `oled_spi.c`。
- 显示驱动更像一个独立模块。

## 7. 为什么采用“中断收字节，主循环解析显示”的软件结构

当前 `main.c` 的核心流程是：

```text
初始化 HAL
初始化系统时钟
初始化 GPIOA 和 USART2
初始化 OLED 显示
重置 NMEA 捕获器
启动 USART2 单字节中断接收
进入 while(1)
    如果有完整 RMC 帧
        解析 RMC
        刷新 OLED
```

UART 回调中只做：

```text
收到 1 字节
    -> BDS_NMEA_PushByte(byte)
    -> 重新启动下一字节中断接收
```

### 7.1 为什么不用主循环轮询串口

轮询方式大致是：

```text
while (1)
{
  不停检查 UART 有没有数据
  有数据就读
}
```

问题是 OLED 刷新、字符串解析、延时等操作都可能让主循环暂时没空读串口。

串口数据是连续来的，如果主循环处理不及时，就可能丢字节。

中断接收的好处：

1. 字节到达后马上进入回调。
2. 主循环可以慢一点处理完整帧。
3. 接收和解析解耦。
4. 代码比 DMA 简单，适合当前数据量。

### 7.2 为什么不在中断里解析整条语句

中断里做的事情越多，越容易影响系统稳定。

如果在中断里做这些事情：

- `strchr`
- `strlen`
- `snprintf`
- 坐标转换
- OLED 刷新

就会导致中断执行时间变长。新的串口字节来了可能来不及处理。

当前设计把工作分成两层：

| 位置 | 做什么 | 原因 |
| --- | --- | --- |
| UART 中断回调 | 收 1 字节、拼帧、标记 ready | 必须快 |
| 主循环 | 取完整帧、解析、刷新 OLED | 可以慢一点 |

这是一种典型的嵌入式设计思路：

```text
中断做最小必要工作。
复杂业务放到主循环。
```

### 7.3 为什么暂时不用 DMA

DMA 可以自动把串口数据搬到内存缓冲区，适合高频、大量、连续数据。

但当前项目：

- NMEA 数据量不大。
- GPS 更新频率通常不高。
- 一次只需要解析 RMC。
- 单字节中断已经足够。
- 初学阶段更容易理解中断流程。

所以不用 DMA 是合理的。

后续如果要提高稳定性，可以扩展为：

```text
USART2 + DMA circular buffer + idle line interrupt
```

但那会增加缓冲区管理复杂度，不是当前阶段必须项。

## 8. 软件分层设计

当前工程已经从“所有代码写在 main.c”改成模块化结构。

可以按 4 层理解：

```text
应用调度层：main.c
业务显示层：bds_display.c/h
协议解析层：bds_nmea.c/h
硬件驱动层：oled_spi.c/h + HAL + MSP/IT
```

### 8.1 `main.c` 的职责

`main.c` 不负责具体协议细节，也不负责 OLED 时序。

它只负责：

1. 初始化 MCU 基础环境。
2. 初始化 USART2。
3. 初始化显示模块。
4. 启动串口中断接收。
5. 在主循环里调度解析和显示。
6. 提供 UART 接收完成回调和错误回调。

这种写法的好处：

- 主流程非常清楚。
- 出问题时能判断是初始化、接收、解析还是显示层的问题。
- 后续增加功能时不至于把 `main.c` 写成几千行。

### 8.2 `bds_nmea.c/h` 的职责

这个模块只关心 NMEA 协议和数据格式。

它负责：

| 功能 | 说明 |
| --- | --- |
| `BDS_NMEA_ResetCapture()` | 丢弃当前半帧，等待下一条 `$` |
| `BDS_NMEA_PushByte()` | 中断收到 1 字节后送入捕获器 |
| `BDS_NMEA_TakeFrame()` | 主循环取出完整 RMC 帧 |
| `BDS_NMEA_ParseFrame()` | 解析 RMC 字段 |
| `BDS_NMEA_FormatCoordinate()` | 把 NMEA 度分格式转十进制度 |
| `BDS_NMEA_FormatBeijingTime()` | 把 UTC 转北京时间 |

它不关心：

- OLED 怎么写字。
- STM32 用的是 USART1 还是 USART2。
- GPS 模块接哪个引脚。

这就是解耦。

### 8.3 `bds_display.c/h` 的职责

这个模块只关心业务显示。

它负责把 `BdsFix` 结构体变成 OLED 上的文字：

| 状态 | 显示 |
| --- | --- |
| 刚开机 | `BDS WAITING` |
| 收到有效定位 | `BDS FIX`、时间、经纬度 |
| 收到 RMC 但未定位 | `BDS NO FIX`、时间、经纬度隐藏 |
| 解析失败 | `BDS PARSE ERR` |

它不关心：

- UART 怎么收。
- NMEA 字段怎么切。
- OLED 每个字节怎么发。

这种设计让显示逻辑更容易改。比如以后想改成中文提示、增加卫星数量、显示海拔，只需要改显示层和解析层，不必碰串口接收。

### 8.4 `oled_spi.c/h` 的职责

这个模块是底层驱动。

它负责：

1. GPIOB 初始化。
2. 模拟 SPI 写字节。
3. 区分命令和数据。
4. SSD1306 初始化。
5. 清屏。
6. 按行写 ASCII。
7. 显示固定中文点阵“作者：张立超”。

它不关心：

- 显示的是 GPS 还是温度。
- 坐标是否有效。
- 什么时候收到串口数据。

底层驱动越纯粹，越容易复用。

### 8.5 MSP 和中断文件的职责

`stm32f1xx_hal_msp.c` 负责硬件资源初始化：

```text
USART2 时钟使能
GPIOA 时钟使能
PA2 配置为 USART2_TX 复用推挽输出
PA3 配置为 USART2_RX 输入
USART2_IRQn 使能
```

`stm32f1xx_it.c` 负责中断入口：

```c
void USART2_IRQHandler(void)
{
  HAL_UART_IRQHandler(&huart2);
}
```

HAL 处理完中断后，会回到 `main.c` 中的：

```c
HAL_UART_RxCpltCallback()
HAL_UART_ErrorCallback()
```

这样中断入口和业务处理分开，符合 HAL 工程的常见结构。

## 9. 数据流完整分析

### 9.1 启动阶段

上电后流程：

```text
HAL_Init()
    -> 初始化 HAL 状态和 SysTick

SystemClock_Config()
    -> 使用 HSI 8MHz
    -> APB1/APB2 都是 8MHz

MX_GPIO_Init()
    -> 打开 GPIOA 时钟

MX_USART2_UART_Init()
    -> 配置 USART2 为 9600 8N1
    -> HAL_UART_MspInit() 配置 PA2/PA3 和中断

BDS_Display_Init()
    -> OLED_Init()
    -> 显示 BDS WAITING 默认页面

BDS_NMEA_ResetCapture()
    -> 清空 NMEA 捕获状态

HAL_UART_Receive_IT(&huart2, &bds_rx_byte, 1)
    -> 启动第一个字节接收
```

### 9.2 接收阶段

GPS/BDS 模块不断输出 NMEA 字节。

每收到 1 个字节：

```text
USART2 硬件收到字节
        -> 触发 USART2_IRQHandler()
        -> HAL_UART_IRQHandler(&huart2)
        -> HAL_UART_RxCpltCallback()
        -> BDS_NMEA_PushByte(bds_rx_byte)
        -> 再次 HAL_UART_Receive_IT()
```

这里有一个非常关键的点：

```text
HAL_UART_Receive_IT() 收 1 字节后会结束。
如果不在回调里重新调用，就只会收到第一个字节。
```

当前代码已经在回调里重新启动下一字节接收，这是正确的。

### 9.3 拼帧阶段

NMEA 语句通常是：

```text
$GNRMC,...\r\n
```

`BDS_NMEA_PushByte()` 的策略：

1. 看到 `$`，认为新帧开始。
2. 后续字节写入捕获缓冲区。
3. 看到 `\n`，认为一帧结束。
4. 判断是不是 RMC：
   - `$GPRMC`
   - `$GNRMC`
   - `$BDRMC`
5. 如果是 RMC，就复制到完整帧缓冲区，并设置 `bds_frame_ready = 1`。
6. 重置捕获状态，等待下一条 `$`。

这个设计的好处：

- 不需要保存所有 NMEA 语句。
- GGA、GSV、VTG 等暂时无关语句会被忽略。
- 主循环只处理对显示最有用的 RMC。
- 收到错乱半帧时，下一次 `$` 可以重新同步。

### 9.4 主循环取帧

主循环不断调用：

```c
BDS_NMEA_TakeFrame(frame, sizeof(frame))
```

如果有新帧：

1. 临时关闭中断。
2. 把共享缓冲区复制到主循环局部变量。
3. 清除 ready 标志。
4. 重新打开中断。

为什么要临时关闭中断？

因为 `bds_frame_buffer` 和 `bds_frame_ready` 同时被中断和主循环访问。

如果主循环复制到一半，中断刚好写入新帧，就可能读到半新半旧的数据。

短暂关闭中断可以保证复制过程完整。

### 9.5 解析阶段

主循环拿到完整 RMC 后调用：

```c
BDS_NMEA_ParseFrame(frame, &fix)
```

解析目标字段：

| 字段 | 说明 |
| --- | --- |
| UTC 时间 | `hhmmss.sss` |
| 定位状态 | `A` 有效，`V` 无效 |
| 纬度 | `ddmm.mmmm` |
| N/S | 北纬/南纬 |
| 经度 | `dddmm.mmmm` |
| E/W | 东经/西经 |

解析后保存到：

```c
typedef struct
{
  char utc[11];
  char status[2];
  char latitude[11];
  char ns[2];
  char longitude[12];
  char ew[2];
} BdsFix;
```

这个结构体的设计很务实：

- 保留原始字符串，避免过早转换。
- 显示时再格式化。
- 不在中断里处理浮点。
- 字段长度固定，适合小 MCU。

### 9.6 显示阶段

解析成功后：

```c
BDS_Display_ShowFix(&fix);
```

显示逻辑：

| 条件 | 显示策略 |
| --- | --- |
| `fix->status[0] == 'A'` | 显示 `BDS FIX`，显示经纬度 |
| 状态不是 `A` | 显示 `BDS NO FIX`，隐藏经纬度 |
| 解析失败 | 显示 `BDS PARSE ERR` |
| 刚启动还没收到 RMC | 显示 `BDS WAITING` |

这几个状态区分得很好，因为它们代表不同故障阶段：

| OLED 状态 | 代表什么 |
| --- | --- |
| `BDS WAITING` | 没收到可识别 RMC |
| `BDS NO FIX` | 收到了 RMC，但卫星还没定位 |
| `BDS FIX` | 定位成功 |
| `BDS PARSE ERR` | 收到疑似 RMC，但字段不符合预期 |

## 10. NMEA/RMC 解析设计

### 10.1 为什么只解析 RMC

NMEA 里有很多语句，例如：

| 语句 | 用途 |
| --- | --- |
| RMC | 推荐最小定位信息，含时间、状态、经纬度、速度等 |
| GGA | 定位质量、卫星数、海拔 |
| GSV | 可见卫星信息 |
| GSA | 定位精度和使用卫星 |
| VTG | 地面速度和航向 |

当前项目的显示目标是：

```text
定位状态
北京时间
纬度
经度
```

RMC 已经包含这些核心字段，所以只解析 RMC 是合理的。

这样做的好处：

- 解析逻辑简单。
- 内存占用小。
- 主循环处理量小。
- 初学者更容易理解。
- 显示内容直接对应 RMC 字段。

后续如果要显示卫星数量、海拔、定位质量，可以再扩展 GGA。

### 10.2 为什么支持 GPRMC、GNRMC、BDRMC

不同模块的 talker ID 可能不同：

| 语句头 | 含义 |
| --- | --- |
| `$GPRMC` | GPS talker |
| `$GNRMC` | 多星座 GNSS talker |
| `$BDRMC` | 北斗 talker |

如果代码只识别 `$GPRMC`，北斗模块输出 `$BDRMC` 或多星座模块输出 `$GNRMC` 时，系统就会一直显示 `BDS WAITING`。

当前工程支持三种 RMC 头，这是对实际模块兼容性的考虑。

### 10.3 坐标为什么要转换

NMEA 经纬度不是十进制度。

纬度格式：

```text
ddmm.mmmm
```

经度格式：

```text
dddmm.mmmm
```

例如：

```text
3000.1234
```

不是 3000.1234 度，而是：

```text
30 度 00.1234 分
```

换算公式：

```text
十进制度 = 度 + 分 / 60
```

当前代码把结果格式化成：

```text
LAT:30.xxxxxxN
LON:120.xxxxxxE
```

这样比直接显示 `3000.xxxx` 更符合用户理解。

### 10.4 为什么不用浮点格式化

在 PC 上写：

```c
printf("%.6f", value);
```

很自然。

但在 STM32 裸机工程里，浮点 `printf` 有几个问题：

1. 可能默认没启用。
2. 启用后会增加 Flash 占用。
3. 对小 MCU 来说没必要。
4. 调试时容易出现格式化异常。

当前代码使用整数定点：

```text
内部用“百万分之一度”保存
显示时拆成 整数部分 + 6 位小数部分
```

好处：

- 不依赖浮点库。
- 输出稳定。
- 资源占用更小。
- 适合 STM32F103 这类小 MCU。

### 10.5 北京时间转换的设计取舍

RMC 语句给的是 UTC 时间。

当前代码显示：

```text
北京时间 = UTC + 8 小时
```

它只显示时分秒，不显示日期。

这意味着：

```text
UTC 23:30:00 -> 北京时间 07:30:00
```

小时会回卷，但日期不会显示加一天。

这是一个合理的简化，因为当前 OLED 显示空间有限，项目目标是显示定位状态和坐标，不是完整时间日历系统。

## 11. OLED 显示设计

### 11.1 页面布局

当前 OLED 使用 128x64 像素，按 SSD1306 page 方式组织。

每页 8 像素高，共 8 页。

当前显示布局：

| 行 | 内容 |
| --- | --- |
| 第 0 行 | `BDS WAITING` / `BDS NO FIX` / `BDS FIX` / `BDS PARSE ERR` |
| 第 1 行 | `TIME:HH:MM:SS` |
| 第 2 行 | `LAT:--` 或 `LAT:xx.xxxxxxN` |
| 第 3 行 | `LON:--` 或 `LON:xxx.xxxxxxE` |
| 第 6-7 行 | `作者：张立超` |

这样的布局有几个好处：

1. 第一行显示状态，最关键。
2. 时间、纬度、经度各占一行，清晰。
3. 经纬度未定位时显示 `--`，避免误导。
4. 作者信息放底部，不影响核心数据。

### 11.2 为什么每行补齐 21 个字符

`bds_display.c` 里有一个细节：

```text
128 像素宽 / 每字符 6 像素 ≈ 21 个字符
```

所以显示层会把每行补成 21 个 ASCII 字符。

原因是：

如果上一行显示：

```text
BDS PARSE ERR
```

下一次只写：

```text
BDS FIX
```

如果不清空或补空格，后面可能残留旧字符。

当前 `BDS_Display_WriteLine()` 统一补空格，保证旧内容被覆盖。

### 11.3 为什么 OLED 驱动只内置有限字库

`oled_spi.c` 内置了：

- 数字。
- 大写字母。
- 冒号。
- 小数点。
- 减号。
- 斜杠。
- 空格。
- 固定中文点阵“作者：张立超”。

没有内置完整中文字库。

原因：

1. 完整中文字库会占用大量 Flash。
2. 当前项目显示内容固定，没必要支持所有汉字。
3. ASCII 足够显示状态、时间、经纬度。
4. 作者信息固定，用点阵数组最省事。

这是资源受限 MCU 中常见的取舍。

## 12. 时钟、中断和外设配置设计

### 12.1 系统时钟为什么用 HSI 8MHz

当前 `SystemClock_Config()` 使用：

```text
HSI 8MHz
PLL 不启用
AHB  = 8MHz
APB1 = 8MHz
APB2 = 8MHz
```

这样做的好处：

- 不依赖外部晶振是否存在。
- 启动简单。
- 配置风险低。
- 9600 串口和 OLED GPIO 模拟 SPI 都够用。

代价是：

- CPU 性能不是最高。
- 如果后续加复杂算法或高速外设，可能需要切到 HSE + PLL 72MHz。

当前项目数据量小，用 8MHz 足够。

### 12.2 USART2 中断优先级

`stm32f1xx_hal_msp.c` 中：

```c
HAL_NVIC_SetPriority(USART2_IRQn, 0, 0);
HAL_NVIC_EnableIRQ(USART2_IRQn);
```

这说明 USART2 中断优先级较高。

当前项目中没有复杂的其他中断，因此这个设置不会造成明显冲突。

如果后续增加定时器、按键、DMA 等中断，需要重新规划优先级。

### 12.3 GPIO 模式

USART2：

| 引脚 | 模式 |
| --- | --- |
| PA2 / TX | 复用推挽输出 |
| PA3 / RX | 输入，无上下拉 |

OLED：

| 引脚 | 模式 |
| --- | --- |
| PB8 / SCK | 推挽输出 |
| PB9 / SDA | 推挽输出 |
| PB12 / DC | 推挽输出 |
| PB13 / CS | 推挽输出 |

这里没有用 I2C，所以 OLED 的 SDA 不需要开漏和上拉。它只是 SPI 数据输入线。

## 13. 当前设计的主要好处

### 13.1 硬件接口简单

主要接线只有：

- GPS/BDS：VCC、GND、TX、RX。
- OLED：VDD、GND、SCK、SDA、DC、CS、RES。
- ST-LINK：SWDIO、SWCLK、GND、可选 3.3V/NRST。

适合面包板和杜邦线实验。

### 13.2 软件职责清楚

模块边界明确：

```text
main.c          调度
bds_nmea        协议
bds_display     显示业务
oled_spi        显示驱动
HAL/MSP/IT      芯片外设
```

这比把所有代码堆在 `main.c` 里更容易维护。

### 13.3 出问题容易定位

可以按现象判断：

| 现象 | 优先怀疑 |
| --- | --- |
| OLED 完全不亮 | OLED 供电、PB8/PB9/PB12/PB13、RES、OLED 类型 |
| 一直 `BDS WAITING` | 没收到 RMC，查 GPS TX、PA3、波特率、模块输出 |
| 一直 `BDS NO FIX` | 串口通了，但卫星未定位，查天线和环境 |
| `BDS PARSE ERR` | 收到疑似 RMC，但帧内容异常 |
| 时间有但坐标没有 | RMC 状态为 V，未定位 |
| 经纬度格式怪 | 坐标转换或原始 NMEA 格式问题 |

状态显示本身就是调试信息。

### 13.4 资源占用低

当前设计没有使用：

- RTOS。
- 动态内存。
- 浮点格式化。
- 大字库。
- 大屏缓存。
- DMA 复杂缓冲。

这让工程更适合 STM32F103C8T6。

### 13.5 后续扩展方便

因为协议、显示、驱动分层，后续可以扩展：

| 想扩展什么 | 主要改哪里 |
| --- | --- |
| 显示卫星数量 | `bds_nmea` 增加 GGA 解析，`bds_display` 增加显示 |
| 换 I2C OLED | 替换 `oled_spi`，尽量不动 `bds_nmea` |
| 加串口日志 | 增加 USART1 日志模块 |
| 加按键切换页面 | 增加 GPIO/EXTI 和显示页面状态 |
| 加 DMA 接收 | 改 UART 接收层，尽量保留解析层 |
| 显示日期 | 扩展 RMC 日期字段解析 |

## 14. 当前设计的限制和风险

好的设计不是没有缺点，而是清楚知道取舍。

### 14.1 `.ioc` 和源码不一致

这是当前工程最需要注意的维护风险。

`.ioc` 里是 USART1/PA9/PA10，但源码实际是 USART2/PA2/PA3。

风险：

```text
如果以后用 CubeMX 重新生成代码，可能覆盖 USART2 相关手写配置。
```

建议后续把 `.ioc` 同步更新为：

```text
USART2 -> PA2/PA3
SYS -> Serial Wire
OLED GPIO -> PB8/PB9/PB12/PB13
```

### 14.2 没有校验 NMEA checksum

当前代码通过 `$` 和 `\n` 捕获帧，并解析字段，但没有验证 `*xx` 校验和。

在干扰较小的短线实验中通常可以工作。

如果用于更可靠的场景，建议增加：

```text
计算 '$' 和 '*' 之间所有字符异或值
与 '*' 后两位十六进制校验值比较
```

这样可以过滤错误帧。

### 14.3 只保留最后一条 RMC

当前 `bds_frame_buffer` 只保存最后一条完整 RMC。

如果主循环处理太慢，新帧可能覆盖旧帧。

当前 GPS 更新慢、OLED 刷新内容少，这不是大问题。

如果后续数据频率提高，可以改成环形队列。

### 14.4 OLED 驱动只适配当前屏

当前初始化序列按 SSD1306 类 OLED 写。

如果换成 SH1106 或其他控制器，可能出现：

- 显示偏移。
- 不亮。
- 命令不兼容。

需要根据屏幕控制芯片调整初始化序列和列偏移。

### 14.5 未显示日期和定位质量

当前只显示：

- 状态。
- 时间。
- 纬度。
- 经度。

没有显示：

- 日期。
- 卫星数量。
- 海拔。
- HDOP。
- 速度。
- 航向。

这是为了保持显示简洁和解析简单。

毕业设计如果需要更完整，可以扩展 GGA 和 RMC 后续字段。

## 15. 从毕业设计角度怎么描述这个方案

可以这样表述设计思路：

```text
本系统以 STM32F103C8T6 为主控，北斗/GPS 模块作为定位数据采集单元，OLED 作为显示单元。北斗/GPS 模块通过 UART 输出标准 NMEA 数据，STM32 使用 USART2 中断方式接收字节流，并在软件中完成 RMC 语句筛选、字段解析、坐标转换和时间转换。显示部分采用 7 针 SPI OLED，由 GPIO 模拟 SPI 实现底层驱动，显示层根据解析结果刷新定位状态、北京时间和经纬度。
```

再进一步说明设计优势：

```text
该方案硬件连接简单、接口标准、软件层次清晰。USART2 保证定位数据接收可靠，GPIO 模拟 SPI 适配现有 OLED 接线，NMEA 解析层和 OLED 显示层解耦，便于后续扩展和故障定位。系统未使用复杂 RTOS 和浮点格式化，降低了资源占用和调试难度，适合 STM32F103C8T6 这类资源有限的单片机。
```

## 16. 推荐的软件开发和调试顺序

不要一上来就整机联调。

推荐顺序：

```text
1. STM32 最小系统能下载
2. OLED 单独显示固定字符串
3. GPS/BDS 模块用 USB-TTL 验证 NMEA 输出
4. STM32 USART2 能收到原始字节
5. 捕获完整 RMC 帧
6. 解析时间和状态
7. 转换经纬度
8. OLED 显示 NO FIX / FIX
9. 室外或窗边验证定位成功
```

每一步都有明确成功标准：

| 步骤 | 成功标准 |
| --- | --- |
| 最小系统 | ST-LINK 可连接，可烧录 |
| OLED | 显示 `BDS WAITING` |
| GPS 单测 | 串口助手看到 `$GNRMC` / `$GPRMC` / `$BDRMC` |
| USART2 | PA3 能收到字节 |
| NMEA 捕获 | 能得到完整 `$...` 语句 |
| RMC 解析 | 能提取时间、状态、经纬度字段 |
| 显示 | OLED 状态随定位变化 |
| 定位 | 显示 `BDS FIX` 和有效坐标 |

## 18. 总结

这个项目的总体设计可以概括为：

```text
用 STM32F103C8T6 做主控。
用 USART2 接北斗/GPS 模块，接收标准 NMEA 定位数据。
用软件模块筛选和解析 RMC 语句。
用整数定点方式转换经纬度，避免浮点格式化。
用 GPIO 模拟 SPI 驱动 7 针 OLED。
用显示层把定位状态、北京时间、经纬度组织成清晰页面。
```

它的设计重点不是“代码写得多”，而是每一层职责清楚：

```text
硬件连接层：确保 PA2/PA3、PB8/PB9/PB12/PB13 接对。
外设配置层：确保 USART2、GPIO、NVIC、时钟可用。
数据接收层：确保串口字节不断流。
协议解析层：确保 RMC 字段正确提取。
数据转换层：确保 UTC 和坐标显示成用户能看懂的格式。
显示输出层：确保 OLED 页面稳定刷新。
调试验证层：通过 WAITING / NO FIX / FIX / PARSE ERR 判断问题位置。
```

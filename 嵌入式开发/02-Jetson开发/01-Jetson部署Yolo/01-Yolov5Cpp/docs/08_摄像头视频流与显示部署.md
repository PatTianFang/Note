# 08 摄像头、视频流与显示部署

这一篇讲输入源。图片版跑通后，再接摄像头。不要一开始就把摄像头、模型、TensorRT、C++ 后处理混在一起排错。

## 输入源类型

常见输入有 4 类：

| 输入源 | 例子 | 难度 |
| --- | --- | --- |
| 图片 | `demo.jpg` | 最简单 |
| 本地视频 | `demo.mp4` | 简单 |
| USB 摄像头 | `/dev/video0` | 中等 |
| CSI 摄像头 | Jetson 摄像头排线 | 中等 |
| RTSP 网络流 | 海康、大华、IPC | 中等到较难 |

推荐顺序：

```text
图片 -> 本地视频 -> USB 摄像头 -> CSI 或 RTSP
```

## USB 摄像头

检查设备：

```bash
ls /dev/video*
v4l2-ctl --list-devices
v4l2-ctl -d /dev/video0 --list-formats-ext
```

C++ 打开方式：

```cpp
cv::VideoCapture cap(0, cv::CAP_V4L2);
```

设置分辨率：

```cpp
cap.set(cv::CAP_PROP_FRAME_WIDTH, 1280);
cap.set(cv::CAP_PROP_FRAME_HEIGHT, 720);
cap.set(cv::CAP_PROP_FPS, 30);
```

如果实际 FPS 不对，以摄像头支持格式为准，不是你设置多少就一定是多少。

## CSI 摄像头

CSI 摄像头优先用 GStreamer。先用命令测试：

```bash
gst-launch-1.0 nvarguscamerasrc ! nvoverlaysink
```

如果要转成 OpenCV 可读取的 BGR：

```bash
gst-launch-1.0 nvarguscamerasrc ! \
  video/x-raw(memory:NVMM),width=1280,height=720,framerate=30/1 ! \
  nvvidconv ! video/x-raw,format=BGRx ! \
  videoconvert ! video/x-raw,format=BGR ! \
  appsink
```

C++ 中使用：

```cpp
std::string pipeline =
    "nvarguscamerasrc ! "
    "video/x-raw(memory:NVMM),width=1280,height=720,framerate=30/1 ! "
    "nvvidconv ! video/x-raw,format=BGRx ! "
    "videoconvert ! video/x-raw,format=BGR ! "
    "appsink drop=1 sync=false";

cv::VideoCapture cap(pipeline, cv::CAP_GSTREAMER);
```

如果打不开，先不要怀疑 YOLOv5。先单独测试 GStreamer pipeline。

## RTSP 网络摄像头

RTSP 常见地址：

```text
rtsp://用户名:密码@摄像头IP:554/路径
```

先用 GStreamer 测试：

```bash
gst-launch-1.0 rtspsrc location=rtsp://user:pass@192.168.1.10/stream latency=100 ! fakesink
```

如果是 H.264，可以尝试硬件解码：

```bash
gst-launch-1.0 rtspsrc location=rtsp://user:pass@192.168.1.10/stream latency=100 ! \
  rtph264depay ! h264parse ! nvv4l2decoder ! nvvidconv ! \
  video/x-raw,format=BGRx ! videoconvert ! video/x-raw,format=BGR ! \
  appsink
```

C++：

```cpp
std::string pipeline =
    "rtspsrc location=rtsp://user:pass@192.168.1.10/stream latency=100 ! "
    "rtph264depay ! h264parse ! nvv4l2decoder ! nvvidconv ! "
    "video/x-raw,format=BGRx ! videoconvert ! video/x-raw,format=BGR ! "
    "appsink drop=1 sync=false";

cv::VideoCapture cap(pipeline, cv::CAP_GSTREAMER);
```

RTSP 排错时要检查：

1. Jetson 能不能 ping 通摄像头。
2. 用户名密码是否正确。
3. 摄像头输出 H.264 还是 H.265。
4. 是否需要主码流或子码流。
5. 延迟参数是否过高。

## 显示和保存

调试时可以显示：

```cpp
cv::imshow("result", frame);
cv::waitKey(1);
```

但显示窗口会影响 FPS。真实部署时可能选择：

1. 不显示，只输出识别结果。
2. 保存报警图片。
3. 推送结果到业务系统。
4. 通过 RTSP/HTTP 输出画面。

第一版建议保留显示，便于肉眼检查。

## 视频循环结构

```cpp
while (true) {
    cv::Mat frame;
    if (!cap.read(frame)) {
        break;
    }

    auto detections = detector.detect(frame);
    drawDetections(frame, detections, labels);

    cv::imshow("YOLOv5 TensorRT", frame);
    if (cv::waitKey(1) == 27) {
        break;
    }
}
```

按 Esc 退出。

## 多线程什么时候再做

第一版不要一上来多线程。单线程流程更容易排错：

```text
读帧 -> 推理 -> 显示
```

等单线程稳定后，再考虑：

```text
线程 1：读帧
线程 2：推理
线程 3：显示或输出
```

多线程能减少等待，但也会带来：

1. 队列积压。
2. 延迟增加。
3. 旧帧处理。
4. 锁和资源释放问题。

工业实时识别通常宁可丢旧帧，也不要排队处理很久以前的帧。

## 本篇验收标准

完成本篇后，你应该能：

1. 单独打开摄像头。
2. 单独播放本地视频。
3. 用 C++ 读取摄像头 frame。
4. 在画面上画检测框。
5. 连续运行 30 分钟不崩溃。

如果 30 分钟内出现卡死、内存上涨、摄像头断流，先解决稳定性，再谈性能。

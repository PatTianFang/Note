# 09 性能优化、测试与常见问题

这一篇讲如何让程序更快、更稳。第一原则：先测量，再优化。

## 不要只看总 FPS

总 FPS 只能告诉你“慢”，不能告诉你“哪里慢”。应该分段测：

```text
读帧时间
预处理时间
推理时间
后处理时间
显示时间
总时间
```

建议日志：

```text
read=4.2ms preprocess=6.8ms infer=18.5ms postprocess=2.1ms display=5.0ms total=36.6ms fps=27.3
```

如果显示时间很高，关闭 `imshow` 后 FPS 可能明显提升。

## Jetson 状态监控

运行：

```bash
tegrastats
```

重点看：

1. GPU 是否在工作。
2. RAM 是否持续上涨。
3. 温度是否过高。
4. 是否降频。
5. CPU 是否被某个步骤打满。

检查电源模式：

```bash
sudo nvpmodel -q
```

测试阶段可以开启较高性能模式，再锁频：

```bash
sudo nvpmodel -m <查询到的最大性能模式编号>
sudo jetson_clocks
```

不同系统镜像模式编号可能不同，先用 `sudo nvpmodel -q --verbose` 确认。

## 优化顺序

推荐按这个顺序优化：

1. 使用 TensorRT FP16。
2. 使用较小模型，例如 YOLOv5s 或 YOLOv5n。
3. 固定输入尺寸和 batch=1。
4. 降低输入尺寸，例如 640 改 512 或 416。
5. 减少显示开销。
6. 优化预处理。
7. 优化后处理。
8. 再考虑 INT8。
9. 再考虑多线程。

不要第一步就做 INT8 或多线程。

## 输入尺寸的影响

YOLOv5 输入尺寸越大，精度可能越好，但速度越慢。

| 输入尺寸 | 特点 |
| --- | --- |
| 416x416 | 快，小目标可能差 |
| 512x512 | 折中 |
| 640x640 | 常用默认，第一版推荐 |
| 1280x1280 | 小目标更好，但 Xavier 上可能慢 |

如果目标很大，例如安全帽、车辆、人员，640 往往足够。若是很小的缺陷，需要重新评估摄像头分辨率、镜头、距离，而不是只增大模型输入。

## 模型大小的影响

| 模型 | 速度 | 精度 | 建议 |
| --- | --- | --- | --- |
| YOLOv5n | 最快 | 较低 | 低功耗、低延迟 |
| YOLOv5s | 快 | 中等 | 第一版推荐 |
| YOLOv5m | 中等 | 更好 | 确认性能后再试 |
| YOLOv5l/x | 慢 | 更好 | 不建议新手首选 |

## 常见问题 1：TensorRT 解析 ONNX 失败

可能原因：

1. ONNX opset 太新。
2. TensorRT 版本较老。
3. 导出时用了不支持的算子。
4. 动态 shape 没有设置 profile。

处理顺序：

1. 用 `--opset 12` 重新导出 ONNX。
2. 去掉 `--dynamic`，先用固定输入。
3. 去掉 `--nms`，让 C++ 自己后处理。
4. 用 `trtexec --verbose` 查看具体失败层。

## 常见问题 2：engine 能生成，但 C++ 加载失败

可能原因：

1. engine 不是在这台 Jetson 上生成的。
2. TensorRT 版本变了。
3. 程序链接到了错误版本的 TensorRT。
4. engine 文件损坏。

处理：

```bash
/usr/src/tensorrt/bin/trtexec --loadEngine=best_fp16_640.engine
```

如果 `trtexec` 都加载失败，先重新生成 engine。

## 常见问题 3：没有检测结果

检查顺序：

1. 模型在 Python 里是否能检测。
2. ONNX 输出 shape 是否正确。
3. 输入是否 RGB。
4. 是否除以 255。
5. 是否 NCHW。
6. `conf_threshold` 是否太高。
7. 类别数量是否写错。
8. `labels.txt` 顺序是否和训练一致。

可以先把阈值调低：

```text
conf_threshold = 0.10
```

如果低阈值也完全没框，通常是预处理或输出解析错。

## 常见问题 4：框位置偏移

优先检查：

1. letterbox 的缩放比例。
2. padding 的左右上下值。
3. 坐标从 640 映射回原图的公式。
4. `cx, cy, w, h` 是否正确转成 `x1, y1, x2, y2`。
5. 是否把宽高顺序写反。

常见反算：

```text
x1 = (x1_model - pad_x) / scale
y1 = (y1_model - pad_y) / scale
x2 = (x2_model - pad_x) / scale
y2 = (y2_model - pad_y) / scale
```

## 常见问题 5：FPS 很低

分段排查：

1. `trtexec --loadEngine` 纯推理是否快。
2. 如果纯推理快，C++ 慢，检查预处理、显示、视频解码。
3. 如果纯推理也慢，检查模型大小、输入尺寸、FP16 是否启用。
4. 用 `tegrastats` 看 GPU 是否工作。
5. 检查是否处于低功耗模式。

## 常见问题 6：摄像头延迟很大

原因可能是队列积压。处理方式：

1. `appsink drop=1 sync=false`。
2. 推理线程只处理最新帧。
3. 降低摄像头分辨率。
4. 使用子码流。
5. 不要让显示窗口成为瓶颈。

## 常见问题 7：运行一段时间后崩溃

检查：

1. 是否每帧重复创建 TensorRT context。
2. 是否每帧重复 `cudaMalloc`。
3. 是否有内存泄漏。
4. 是否忘记释放 `cv::VideoCapture`。
5. 是否有输出数组越界。
6. 是否动态 shape 下 buffer 大小不匹配。

原则：

```text
初始化时分配资源
循环中重复使用资源
退出时释放资源
```

## INT8 是否值得做

INT8 可以更快，但需要校准数据，并且可能损失精度。建议满足下面条件后再做：

1. FP16 已经跑通。
2. FPS 仍不够。
3. 有代表性的校准图片。
4. 能接受并验证精度变化。

第一版不要做 INT8。

## 本篇验收标准

完成本篇后，你应该有：

1. 分段耗时日志。
2. `tegrastats` 监控结果。
3. FP16 engine。
4. 至少 30 分钟稳定运行测试。
5. 明确知道瓶颈在哪个步骤。

# 14 部署运维与故障排查手册

这一篇讲程序跑起来之后怎么长期运行、怎么记录日志、怎么处理断流和现场故障。

## 调试版和部署版要分开

调试版特点：

1. 显示窗口。
2. 打印较多日志。
3. 可以手动退出。
4. 参数可以写死。

部署版特点：

1. 不一定显示窗口。
2. 日志可追踪。
3. 摄像头断开能重连。
4. 程序崩溃能自动重启。
5. 模型路径、阈值、摄像头地址来自配置文件。

不要把调试版直接当生产版长期运行。

## 建议配置文件

可以使用简单 YAML 或 JSON。示例：

```yaml
engine: models/best_fp16_640.engine
labels: models/labels.txt
input_width: 640
input_height: 640
num_classes: 3
conf_threshold: 0.25
nms_threshold: 0.45
source: usb:0
display: true
save_alarm_image: false
log_interval_frames: 30
```

第一版如果不会解析 YAML，可以先用命令行参数。等程序稳定后再加配置文件。

## 日志应该记录什么

启动时记录：

1. 程序版本。
2. engine 路径。
3. labels 路径。
4. 输入尺寸。
5. 类别数。
6. 阈值。
7. 摄像头源。
8. TensorRT engine 输入输出 shape。

运行时定期记录：

```text
fps
read_ms
preprocess_ms
infer_ms
postprocess_ms
display_ms
detection_count
```

异常时记录：

1. 摄像头打不开。
2. 读帧失败。
3. engine 加载失败。
4. CUDA 错误。
5. 输出 shape 不匹配。

## 断流重连策略

摄像头或 RTSP 可能断流。建议策略：

```text
读帧失败
  -> 释放 VideoCapture
  -> 等待 1 到 3 秒
  -> 重新打开
  -> 连续失败超过 N 次后输出严重日志
```

伪代码：

```cpp
int fail_count = 0;

while (running) {
    cv::Mat frame;
    if (!cap.read(frame)) {
        fail_count++;
        cap.release();
        std::this_thread::sleep_for(std::chrono::seconds(2));
        cap.open(source);
        continue;
    }

    fail_count = 0;
    // detect
}
```

不要读帧失败就直接退出，除非这是调试程序。

## systemd 开机启动

创建服务文件：

```bash
sudo nano /etc/systemd/system/yolov5-detector.service
```

示例：

```ini
[Unit]
Description=YOLOv5 TensorRT C++ Detector
After=network.target

[Service]
WorkingDirectory=/home/nvidia/YoloJetsonCpp
ExecStart=/home/nvidia/YoloJetsonCpp/build/yolo_detect --config /home/nvidia/YoloJetsonCpp/config.yaml
Restart=always
RestartSec=3
User=nvidia

[Install]
WantedBy=multi-user.target
```

启用：

```bash
sudo systemctl daemon-reload
sudo systemctl enable yolov5-detector
sudo systemctl start yolov5-detector
```

查看状态：

```bash
sudo systemctl status yolov5-detector
journalctl -u yolov5-detector -f
```

调试阶段不要急着开机启动。先手动跑稳定，再交给 systemd。

## 模型替换流程

替换模型时，不要只覆盖 engine。建议流程：

1. 停止服务。
2. 备份旧 engine、ONNX、labels。
3. 拷贝新 ONNX 和 labels。
4. 在 Jetson 本机重新生成 engine。
5. 手动运行程序测试图片。
6. 手动运行程序测试摄像头。
7. 确认无误后启动服务。

命令示例：

```bash
sudo systemctl stop yolov5-detector
cp models/best_fp16_640.engine models/best_fp16_640.engine.bak
/usr/src/tensorrt/bin/trtexec --onnx=models/best.onnx --saveEngine=models/best_fp16_640.engine --fp16 --workspace=2048
./build/yolo_detect --engine models/best_fp16_640.engine --image test_images/demo.jpg
sudo systemctl start yolov5-detector
```

## 现场排障顺序

如果现场说“识别程序不工作”，按这个顺序查：

1. Jetson 是否开机。
2. 服务是否运行。
3. 日志里是否有错误。
4. 摄像头是否在线。
5. engine 是否存在。
6. `trtexec` 是否能加载 engine。
7. 图片版是否还能识别。
8. 视频流是否能单独打开。
9. `tegrastats` 是否显示过热或内存耗尽。
10. 最近是否换过模型、摄像头、系统或网络。

不要第一步就改代码。

## 常见现场问题

### 服务一直重启

查看：

```bash
systemctl status yolov5-detector
journalctl -u yolov5-detector -n 100
```

常见原因：

1. 工作目录不对。
2. engine 路径不对。
3. 用户权限不够。
4. 摄像头权限不够。
5. 缺少环境变量。

### 手动运行正常，systemd 运行失败

systemd 环境变量比终端少。解决：

1. 使用绝对路径。
2. 设置 `WorkingDirectory`。
3. 确认 `User`。
4. 必要时在服务文件中设置环境变量。

### 摄像头偶发断开

处理：

1. 程序加重连。
2. 检查 USB 供电。
3. 降低分辨率和 FPS。
4. RTSP 使用子码流。
5. 网络摄像头检查交换机和网线。

### 磁盘满

如果保存报警图片或日志，必须限制数量。

建议：

1. 日志按天滚动。
2. 报警图片按目录分日期。
3. 设置最大保存天数。
4. 定期清理旧文件。

## 最低交付标准

交付前至少满足：

1. 图片识别正确。
2. 摄像头识别正确。
3. 连续运行稳定。
4. 日志能定位问题。
5. 服务能停止、启动、重启。
6. 模型替换流程清楚。
7. 现场人员知道怎么查看状态。
